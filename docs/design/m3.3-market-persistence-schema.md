## 1. S1 — Identity Schema（确认 Phase 1.5，不变）

四层身份模型保持，全 surrogate 主键，承接 Phase 1.5 §10.3 冻结：

```
company ──< listed_entity ──< security ──< listing
```

```
company
  company_id        PK  (surrogate, new_internal_id)
  name
  region
  aliases           json
  external_ids      json     ← 已存在；命名空间外部码如 "tushare:ts_code:600519.SH" 落这里
  created_at / updated_at

listed_entity
  listed_entity_id  PK
  company_id        FK -> company
  disclosure_name
  source_ref        json
  created_at / updated_at

security
  security_id       PK
  listed_entity_id  FK -> listed_entity
  ts_code           属性/查找键（UNIQUE 索引，非 identity）
  symbol / name / exchange / security_type / currency
  created_at
  -- external_ids: ❌ Deferred（不进 Phase 2）

listing
  listing_id        PK
  security_id       FK -> security
  exchange / board / listed_at / delisted_at / status

industry
  industry_id       PK
  taxonomy / name / code / level

company_industry
  company_id FK / industry_id FK
```

**冻结不变量**：
- `ts_code` 永远是 `security` 的属性/查找键，**永不是 identity**，永不 = `company_id`。
- `Security.external_ids`：**Deferred**。出现真实多 vendor code（EastMoney / Wind / Bloomberg）
  再抽象，现在不为未来扩 schema。
- 命名空间外部码当前在 `Company.external_ids`（实证：`tushare:ts_code:600519.SH`）。

## 2. S2 — Fact / Observation Schema（本次核心决策）

### 2.1 决策回顾：为何选 C（既非 A 也非纯 B）

**Option A — 三张独立 fact 表（plan §3.2 原写）：不选。**

```
market_fact / financial_fact / announcement_fact / quote_snapshot   （各一张表）
```

- 适合「一个业务系统存不同类型数据」，但 ShanHai 不是行情系统，是「AI 对资本市场世界的持续认知」。
- timeline 会退化成 `union market_fact ∪ financial_fact ∪ announcement_fact ∪ …`，每加一种知识类型
  就加一个 projection——长期越来越像 ETL 数据仓库，而不是知识系统。

**Option B — 单一 `knowledge_observation` 超表：不选（但其认知直觉正确）。**

```
knowledge_observation
  id / logical_key / fact_type / subject_id / predicate
  object_value / captured_at / content_hash / source_ref
```

- subject/predicate/object/time/source 是对的认知模型；但把 `metric_value: float` 压成
  `object_value TEXT` 会让 `CAST(object_value AS NUMERIC)` 遍地出现——分析查询（8 年收入 CAGR /
  毛利率趋势 / ROE 排名 / 同业比较）退化。这是反模式。

**Option C — Knowledge Observation Spine + Typed Observation Detail：✅ Accepted。**

一张统一的 **append-only 观测主干**（承载所有 fact family 的共性 + Temporal 语义），
加**按类型的明细表**（保留结构化字段、1:1 挂在主干上）：

```
                  Knowledge Observation Spine
                           │
              ┌────────────┼─────────────┐
              │            │             │
        Financial       Quote       Announcement
        Detail          Detail       Detail
```

```
knowledge_observation            ← 唯一的 append-only 认知时间轴主干；每条观测一行
  observation_id    PK  (surrogate, immutable, append-only)
  logical_key           Assertion 身份（= 现确定性 fact_id 降级而来，如
                        "financial:{security_id}:2025Q4:revenue" / "maotai.revenue.2025Q4"）
  fact_type             quote / financial / announcement / profile / industry / ...
  subject_type / subject_id        指向 surrogate 身份（company/security/...）
  predicate             如 financial.revenue / closing_price / classified_in_industry
  object_type           object_value 的类型标注（numeric / text / enum / ref ...）
  object_value          规范/展示值（cognition 与 timeline 读这一列的统一投影）
  occurred_at / published_at / captured_at      ← 三时态永不塌缩（Phase 1.5 §10 冻结）
  content_hash          对语义字段取 hash（幂等去重键）
  confidence
  source_ref            json（provider/dataset/raw_snapshot_ref/version/hash）
  schema_version
  created_at            行写入时间（≠ captured_at；纯存储审计）

financial_observation              ← fact_type=financial 时 1:1
  observation_id  FK -> knowledge_observation (PK 同值)
  metric_name / report_period / report_type
  metric_value / unit / currency / yoy / qoq / restated

quote_observation                  ← fact_type=quote 时 1:1（行情不塞 object_value）
  observation_id  FK
  trade_date / open / high / low / close / previous_close / volume / amount

announcement_observation           ← fact_type=announcement 时 1:1
  observation_id  FK
  announcement_id / announcement_type / title
  publish_date / document_url / document_hash(content_ref) / extracted_summary
```

收益：
- **统一的 Temporal 主干**：append-only / content_hash 幂等 / captured_at 历史 / timeline 投影
  全部只面对 `knowledge_observation` 一张表（拿到 Option B 的全部好处）。
- **subject/predicate/object/time/source 跨所有 fact family 一致**（满足 AI cognition 框架）。
- **结构化字段不丢**：财务的 `metric_value`/`yoy`、行情的 OHLCV 留在明细表，分析查询仍走 SQL
  类型列（堵住 Option B 的致命缺口）。
- **领域模型 1:1 映射**：`MarketFact` 共性字段 → 主干；`FinancialFact/AnnouncementFact/
  QuoteSnapshot` 的专有字段 → 各自明细表。新增 fact family 多数只写主干、按需加明细。

> 本质是「single-table inheritance（B）vs class-table inheritance（C）」之争。对一个既要
> **统一认知**、又要**结构化分析**的 Temporal Knowledge Store，**C 是诚实答案**。
> **采纳 C 即改写 plan §3.2 的「三张独立表」表述**（plan §3.2 已加 superseded 注记）。

> **两点与你 schema 草图的对账（忠实记录，不静默改）**：
> 1. 你草图写了 `effective_at`，且只列了 `occurred_at / effective_at / captured_at`。Phase 1.5 §10
>    冻结的三时态是 `occurred_at / published_at / captured_at`，且 [models.py](../../services/market-data/shanhai_market_data/models.py)
>    三类 fact 都已有这三列。本文**保留已冻结的三时态不动**；`effective_at`（事实「生效/在市场可见」
>    的时刻）语义上**等同 `published_at`**，故不新增第四列、用 `published_at` 承载。若你认为
>    `effective_at` 是独立于 published 的第四语义，请在 Phase 2 Implementation Review 指出，我再加列。
> 2. 列名统一用 **`logical_key`**（= Phase 1.5 prose 里的 `logical_fact_key`，= 现 `fact_id`）。
>    §2.2 与下文 S5 同步用此名。

### 2.2 append-only / 幂等的 schema 落点（承接 Phase 1.5 §10.1）

在 `knowledge_observation` 主干上：

```
PRIMARY KEY  : observation_id                  (surrogate, immutable, append-only)
UNIQUE       : (logical_key, content_hash)       ← 幂等：同值重抓不新增行
INDEX        : (logical_key, captured_at)         ← timeline 取「最新观测」
INDEX        : (subject_id, fact_type)            ← 按主体/类型检索
```

`logical_key`（= 现 `fact_id`）从「物理 PK」降为「同一 Assertion 的血缘键」；物理 PK 是观测。详见 §4a (S5 Observation Identity)。

> 实证差距（Phase 2 实现时处理，本阶段不写代码）：今天
> [fact_mapper.py](../../services/market-data/shanhai_market_data/fact_mapper.py) 的 `fact_id`
> 不含 captured_at、[store.py](../../services/market-data/shanhai_market_data/store.py) 按
> `fact_id` 覆盖写；三类 fact 模型**已自带 `captured_at`**，缺的是 `content_hash` 填充
> （现 `SourceRef.hash=None`）。

## 4a. S5 — Observation Identity（新增冻结项，schema 最关键的一点）

身份模型从「fact 即一条记录」升级为「logical fact 拥有多个 immutable observation」：

```
logical_key  (= 现 fact_id 降级而来)
     │
     ├──  observation v1   (observation_id=001, value=170, hash=A)
     ├──  observation v2   (observation_id=002, value=172, hash=B)
     └──  observation v3   (observation_id=003, value=172, hash=B)  ← hash 同 v2 → 幂等去重，不新增行
```

- `observation_id`：**immutable PRIMARY KEY**（surrogate，append-only，写入后永不变更）。
- `fact_id` → **降级为 `logical_key`**：不再是物理主键，只是「同一 Assertion 的血缘键」。
- 去重：**`UNIQUE(logical_key, content_hash)`** —— 同值重抓命中既有行、不新增（v3 场景）；
  值变化（170→172）hash 不同 → 新增一个 observation（v1/v2 并存，承载「市场认知发生过变化」）。

```
冻结：
  PRIMARY KEY  observation_id            (immutable)
  UNIQUE       (logical_key, content_hash)
```

> 这一条把 Phase 1.5 §10.1 的语义落成了不可动摇的身份契约：**observation 不可变、append-only；
> 演化靠新增 observation 表达，而非 update 覆盖**。Phase 2 实现必须服从，不得回退成按 logical_key 覆盖。

## 7. 登记（不实现）：Temporal Knowledge Store 的长期形态

§2 的 Option C 主干 + Phase 1.5 的 append-only/captured_at，是为支撑 ShanHai 的核心查询：

> 「2025 年 3 月时，市场认为贵州茅台是什么状态？」= `knowledge state at time T`，
> 而非 `current company state`。

实现路径（登记，本阶段不做）：基于 `knowledge_observation(logical_fact_key, captured_at)`
做「截至 T 的最新观测」投影，即可重建任意时点的知识快照。这与 M3.4 的 RawSnapshot/provenance
合起来回答「我们为什么认为当前事实是什么、这个认知如何演化」。
