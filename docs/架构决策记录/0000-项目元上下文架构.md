# ADR 0000：ShanHai 项目元上下文架构（ShanHai Meta Context Architecture）

状态：已采纳
日期：2026-06-24

- 正式名称：ShanHai Project Context System
- 简称：Project Context Layer
- 目录：`.shanhai-meta/`
- 入口：根目录 `AI_CONTEXT.md`

> 术语纪律：本层系统级名称**不使用 "Memory"**。"Memory" 保留给 Runtime 世界
> （ADR 0012 Agent Runtime Memory / Experience Memory）。Meta Context 不引入第三种 Memory。
>
> 定位：本 ADR 是所有 ADR（0001+）的**前置基础设施**，而非业务能力的后续演进。

## 背景

ShanHai 采用三方协作模式（Human Owner / Architecture Advisor / Coding Agent），
开发方式为「探索 → 讨论 → 收敛 → ADR → 实现」。但一个结构性缺陷已暴露：

> 项目的知识、决策、架构演进存在于「人的聊天历史」里，而不在「项目本身」。

由此产生的失效：

- 换模型（GPT → Claude → Gemini）→ 丢上下文
- 换平台 / 换办公环境 → 丢上下文
- 新会话 → 从零重新解释项目
- 不同 AI（Trae / Claude / GPT）→ 各自理解不一致
- 长周期开发 → 架构决策逐渐漂移、重复犯已否决的错

聊天平台的额度 / 降级限制只是触发器，**根因是项目缺少一等公民的「上下文层」**。

这与 ShanHai 自身理念高度同构：ShanHai 为 Agent 建 Runtime Experience Memory；
本层为「人 + AI 的工程协作」建 Meta Context，二者正交对称。

## 待决问题

1. 上下文层是业务模块（`services/`）还是元基础设施？
2. 目录如何与代码模块解耦，且不与 `services/memory`、ADR 0009 运行目录、Experience 混淆？
3. 原始对话 / 决策 / 快照如何分层，避免摘要覆盖原文？
4. 导出脏数据（无 role、媒体丢失、死循环噪音）如何处理？
5. 后续对话 / 决策如何持续、幂等地同步进来（多模型、多会话）？
6. AI 启动如何「一句话进入状态」？

## 决定

### D1：定位为元上下文层，不进入 services/

Context Layer **不是 ShanHai Runtime 能力**，不被任何 service import，不参与 Agent 运行。
它属于 Engineering Meta Layer。

```
禁止：services/context-*
正确：仓库根的独立元目录 .shanhai-meta/ + tools/context 脚本
```

### D2：Runtime ↔ Meta 双层对称，独立目录 `.shanhai-meta/`

```
Runtime 世界（给 Agent）                 Meta 世界（给 人 + AI 协作）
  services/{memory,experience,evolution}   .shanhai-meta/{context,decisions,…}
  .shanhai/  运行落盘（可丢弃，gitignore，见 ADR 0009）
  解决：Agent 如何记忆经验？               解决：AI 如何理解项目？
```

目录名定为 `.shanhai-meta/`，**不复用 `.shanhai/`**：后者已被 ADR 0009 占用为
local-first 运行落盘目录（`.shanhai/runs.db`），语义为「可丢弃运行数据、已 gitignore」，
与本层「必须永久版本化的核心协作资产」正相反。复用会引发三重冲突：

1. **语义反转**：disposable runtime data ↔ persistent meta asset；
2. **Git 忽略冲突**：`.shanhai/` 整目录已被忽略，塞入需脆弱的反忽略例外或误纳运行数据；
3. **违反 ADR 0009 边界**：ADR 0009 已立「真正资产不进 `.shanhai/`」。

`.shanhai-meta/`（版本化跟踪）与 `.shanhai/`（运行时忽略）在文件树上一眼可辨。

### D3：三层语义

```
Layer 1  Raw Conversation Archive   原始对话（不改 / 不删 / 永久）      —— AI 历史原料
Layer 2  Decision Registry          为什么这么决定 / 什么被否决          —— AI 最需要
Layer 3  Context Snapshot           当前版本 / 架构 / 冻结 / 阶段（可重建）—— AI 启动入口
```

Summary / Decision 永不替代 Raw。

### D4：ContextEvent 作为统一事实流

不直接从 conversation 生成 snapshot。在 Raw 与派生物之间插入统一中间事实流：

```
Raw Archive（不可变）
   │ import（忠实搬运，不推断 speaker）
   ▼
ContextEvent Stream  ← 单一事实来源（append-only）
   │   type ∈ conversation | decision | review | approval | implementation
   ├── 人工确认 ─→ Decision Registry（Layer 2）
   └── build ───→ Context Snapshot（Layer 3，派生可重建）
```

> **ContextEvent ≠ Runtime Event。** 它只是 append-only 的**工程记录**（engineering record），
> 不参与任何 Agent 运行、不被 service 消费，**严格区别于 ExperienceEvent**（ADR 0014）。

### D5：Role inference 降级

不自动推断 speaker。原则：**宁可 unknown，不错误记忆。**

- import 忠实保留原文与 `raw#id`，`actor` 默认 `unknown`，不做奇偶 / 启发式推断；
- 客观可见的脏数据如实标记（如文本含 `media pointer` → `has_lost_media: true`），
  但「谁说的 / 是否粘贴日志」一律不臆测；
- Raw 永久保存；身份与决策归属由 Decision Registry 人工确认。

### D6：零依赖 / Local-first / 纯文本

脚本仅用 Python 标准库（呼应 ADR 0009 的零依赖调性）。
不引入 Vector DB / Embedding / RAG / 自动知识提取。
存储为 Markdown + JSONL + YAML，Git diff 友好、跨模型可读。

### D7：AI 协作入口

根目录新增 `AI_CONTEXT.md`，定位为 **ShanHai AI Entry Point**（非「AI 上下文文件」），含 Read order：

```
1. AI_CONTEXT.md
2. .shanhai-meta/context/current-state.md
3. .shanhai-meta/decisions/
4. docs/架构决策记录/
5. source code
```

`AGENTS.md` 增量追加一行入口指引（read-diff-append，不覆盖原文）。

### D8：版本控制

`.shanhai-meta/` 纳入 git 跟踪。在 `.gitignore` 显式声明纳管
（消歧：`.shanhai/` 不匹配 `.shanhai-meta/`，技术上无需「取消忽略」，此为防未来误伤的显式声明）。

### D9：Source of Truth 与 Projection 分离

```
事实源（append-only，人工/脚本写入，可回溯）        派生（builder 重建，AI 不直接改）
  conversations/raw/        原始导出                  context/current-state.md
  events/stream.jsonl       ContextEvent 流            context/architecture-summary.md
  decisions/                Decision Registry          context/timeline.md
```

规则：

- 事实源 append-only；
- `context/` 下所有快照**必须由 builder 从事实源重建**；
- **AI 不直接修改 snapshot**（要改，改事实源 + 重跑 builder）。

## 原因

- 用「不可变 Raw + 统一 ContextEvent 事实流 + 派生快照」三段式，既保留全部原料，
  又让 AI 启动只需读极小的 snapshot，呼应 ShanHai 自身 append-only + projection 的设计哲学。
- `.shanhai-meta/` 独立目录避免了与运行时 `.shanhai/`、业务 `services/` 的全部语义冲突。
- 零依赖纯文本契合 local-first 与跨模型可读，无需任何外部服务即可在任意环境恢复上下文。

## 影响

- 新增根级元目录 `.shanhai-meta/` 与 `tools/context/` 脚本，对 `services` / `packages` / `apps`
  **零影响、零新依赖**。
- 三方协作从「人肉复制上下文」升级为「AI 读取统一上下文层」。
- ShanHai 同时拥有两层记忆：Runtime 给 Agent 的 Experience Memory + Meta 给协作的 Context Layer，
  未来可在 ADR 层明确区分（ADR 0012 Runtime Memory ↔ ADR 0000 Meta Context）。
- 后续每个重要讨论收敛后产出一条 ContextEvent / Decision，成为协作流程的一部分。

## 边界（明确不做）

禁止进入：`services/` / `memory/` / `experience/` / agent runtime。

明确延期：Vector DB / Embedding / RAG / 语义检索 / 自动知识提取 / 自动总结替代原文 /
与 Runtime Memory 集成 / 多模型 adapter 差异化提示（`adapters/`）。

## 实施计划（5 Commits，逐个过 Review Gate）

1. **Commit 1 Context Foundation**：`.shanhai-meta/` 骨架 + `AI_CONTEXT.md` +
   `tools/context` skeleton + 本 ADR + AGENTS.md 增量 + .gitignore 纳管声明。
2. **Commit 2 Context Schema**：ContextEvent / Record schema + `stream.jsonl` contract。
3. **Commit 3 Raw Import**（拆 3A/3B）：
   - **3A Raw Archive**：原始导出原封不动迁入 `conversations/raw/`（不解析/不转换/不修改），先建立 Raw Source of Truth；
   - **3B Importer**：schema 稳定后再做 Raw → ContextEvent stream（`actor=unknown`，幂等）。
4. **Commit 4 Decision Registry**：`decision-log` / `rejected-decisions` + 首批决策提炼 + 单条追加入口。
5. **Commit 5 Context Builder**：事实源 → `context/*.md` 派生（幂等可重跑，死循环折叠）。

## 备选方案（已考虑）

- **A：`docs/chat-history/*.md` 堆原始聊天** —— 无结构、无决策层、无法幂等增量，与 `docs/`
  策展文档混淆。否决。
- **B：`services/context-memory` 运行时模块** —— 违反「不进 services」「AI 不直连」边界，
  把元上下文误升为业务能力。否决。
- **C：复用 `.shanhai/` 运行目录** —— 与 ADR 0009 语义 / gitignore 冲突（见 D2）。否决。
- **D（采纳）：`.shanhai-meta/` 元目录 + `tools/context` 脚本 + `AI_CONTEXT.md` 入口 + ContextEvent。**
