## 1. PR-3 的核心问题

PR-3 是第一次触碰现有运行身份语义的阶段。它不是新增模型字段，而是一次 **Identity ownership inversion**。

当前：

```text
RunStore.save_run()
        ↓
RunStore creates run_id
```

目标：

```text
RuntimeKernel.create()
        ↓
run_id generated by Runtime
        ↓
RuntimeContext.identity_context.run_id
        ↓
RunStore persists with external run_id
```

迁移方向：

```text
Storage owns identity
        ↓
Runtime owns identity
```

这会影响后续：

- RuntimeContext trace
- RuntimeEvent trace
- Evaluation trace
- Experience Evolution trace
- Replay / audit

因此 PR-3 必须先冻结 contract，再写实现。

### 1.1 PR-3 补充冻结项

本 Gate 冻结 4 个约束：

| ID | 约束 | 结论 |
|---|---|---|
| R1 | run_id 生成格式不绑定 RunStore | 禁止 `RunStore.generate_run_id()`；identity 格式属于 Runtime 侧 contract。 |
| R2 | RunStore 降级为 trace persistence | RunStore 不再拥有 identity / lifecycle，只负责持久化执行 trace。 |
| R3 | migration window 必须有终点 | `run_id=None` 只允许临时存在，需 warning + deadline + future Gate 关闭。 |
| R4 | PR-3 不改 AgentRunner 行为 | 禁止把 identity migration 与 orchestration migration 混在一个 PR。 |

## Q1. run_id 生命周期

### Q1.1 推荐生命周期

```text
RuntimeKernel.create()
        ↓
run_id generated
        ↓
RuntimeHandle(run_id, state=CREATED)
        ↓
RuntimeContext.identity_context.run_id
        ↓
RuntimeEvent.run_id
        ↓
RunStore.save_run(run, run_id=run_id)
        ↓
RunRecord.run_id
```

### Q1.2 生成位置

目标状态：

```text
Runtime Kernel owns run_id generation
```

禁止目标状态：

```text
RunStore.save_run()
        ↓
generate id
```

### Q1.3 PR-3 是否实现 Kernel.create？

本 Gate 的默认建议：

- PR-3 **不实现 RuntimeKernel.create() 的完整行为**。
- PR-3 只让 RunStore 接受外部 `run_id`，为 PR-5 的 Kernel identity 串联铺路。
- 若需要一个 helper / test fixture 产生 run_id，只能放在测试或 runtime-kernel 内部纯函数，不能把 RuntimeKernel 与 RunStore 在 PR-3 里接起来。

原因：

- PR-3 的目标是 RunStore identity migration，不是 E2E wiring。
- Kernel.create → RuntimeContext → RunStore 串联留 PR-5。

### Q1.4 run_id 格式不绑定 RunStore

禁止：

```python
RunStore.generate_run_id()
```

推荐方向：

```python
RuntimeKernel.create()
        ↓
RunIdentity.create()
        ↓
RuntimeIdentity(run_id="...")
```

或者等价的 runtime-kernel 内部 identity helper。

约束：

- `run_id` 格式属于 Runtime identity contract，不属于 storage implementation。
- RunStore 不决定 run_id 格式。
- RunStore 不知道本次运行是 local execution / distributed execution / replay execution / evaluation replay。
- PR-3 若需要测试 external run_id，只使用固定字符串 fixture，不引入真实 identity generator。

理由：

```text
runtime-kernel
      ├── local execution
      ├── distributed execution
      ├── replay execution
      └── evaluation replay
```

这些执行模式共享 execution identity；identity 不能归属于某个 storage backend。

## Q2. RunStore API compatibility

### Q2.1 当前 API

```python
class RunStore(ABC):
    def save_run(self, run: RunResult) -> str:
        ...
```

当前行为：

```text
save_run(run)
        ↓
store mints run_id
        ↓
returns run_id
```

### Q2.2 目标 API（migration window）

推荐 PR-3 采用兼容窗口：

```python
def save_run(self, run: RunResult, run_id: str | None = None) -> str:
    ...
```

行为：

```text
if run_id is provided:
    persist using external run_id
    return same run_id

if run_id is None:
    mint run_id internally
    emit DeprecationWarning
    return generated run_id
```

### Q2.3 为什么保留 `run_id=None`

原因：

- 保持既有 `AgentRunner._persist` 与旧测试不破。
- PR-3 是 behavior change，但应最小化 blast radius。
- 关闭兼容窗口应另开 Gate，不在 PR-3 一步到位。

### Q2.4 migration window 必须带的治理

如果允许 `run_id=None`，必须同时满足：

- emit `DeprecationWarning`
- docstring 标注 migration window only
- tests 覆盖：
  - `save_run(run)` 仍可用
  - `save_run(run)` 触发 warning
  - `save_run(run, run_id="x")` 返回 `"x"`
  - `get_run("x").run_id == "x"`
  - external run_id path 不再内部 mint
- PROJECT_STATE / CHANGELOG 标注 window 临时存在

### Q2.5 migration deadline

建议冻结为：

> `run_id=None` 兼容分支只保留到 Runtime identity E2E 完成并通过 Review 后；关闭窗口另开独立 Gate。

不在 PR-3 关闭窗口。

### Q2.6 migration phases

PR-3 必须把 migration window 写成阶段性临时状态，而不是永久 API。

```text
Phase 1（PR-3）
    save_run(run, run_id=None)
    allowed
    emits DeprecationWarning
    docs mark migration window

Phase 2（Runtime identity E2E 后）
    save_run(run, run_id: str)
    required in new Runtime path
    old path still only under explicit compatibility tests

Phase 3（future Gate）
    remove fallback
    save_run(run, run_id: str) only
```

禁止长期状态：

```text
RuntimeKernel:
    identity owned by Runtime

RunStore:
    still silently mints identity
```

如果 `run_id=None` 分支长期无 warning / deadline，会导致 identity ownership 重新漂移回 storage。

## Q3. trace identity 一致性

### Q3.1 必须保证的等式

后续完整链路必须满足：

```text
RuntimeContext.identity_context.run_id
        =
RuntimeEvent.run_id
        =
RunRecord.run_id
```

这是 runtime trace 的根。

### Q3.2 PR-3 能测试什么

PR-3 尚不做 E2E，因此只能测试 RunStore 侧：

```python
external_run_id = "run_001"
returned = store.save_run(run, run_id=external_run_id)
record = store.get_run(external_run_id)

assert returned == external_run_id
assert record.run_id == external_run_id
```

PR-3 不应测试 RuntimeContext ↔ RuntimeEvent ↔ RunStore 全链路；那属于 PR-5。

### Q3.3 PR-3 必须避免的错觉

不要在 PR-3 里为了证明全链路而引入：

- RuntimeKernel.execute
- AgentRunner adapter
- RuntimeEventStore
- Experience Runtime
- E2E test

PR-3 只证明：RunStore 可以接受 Runtime-owned identity。

### Q3.4 RunStore 的目标职责

迁移后，RunStore 不负责：

```text
create identity
allocate identity
decide lifecycle
```

RunStore 只负责：

```text
persist execution trace
```

目标抽象：

```text
Runtime Kernel
      │
      │ RuntimeEvent / RunResult
      ▼
Execution Trace Store
```

这与 Runtime Kernel 的边界一致：

```text
orchestrator ≠ executor
runtime identity ≠ storage identity
trace persistence ≠ lifecycle ownership
```

- RunStore identity migration

## Q6. Open Decisions

### D1. PR-3 是否允许 `run_id=None`？

推荐：**允许，但必须 warning + migration window 标注**。

理由：

- 保持旧调用兼容。
- 隔离 behavior change。
- 关闭窗口另开 Gate 更稳。

### D2. PR-3 是否同步修改 AgentRunner？

推荐：**否**。

理由：

- AgentRunner adapter / RuntimeContext 接入属于后续 wiring。
- PR-3 只迁移 RunStore identity contract。

### D3. PR-3 是否同步修改 RuntimeKernel.create？

推荐：**否**。

理由：

- Runtime owns identity 是目标，但 PR-3 只让 Store 接受 external identity。
- Kernel.create 的实际串联留 PR-5，避免提前 E2E。

### D4. PR-3 是否修改 persistent stores？

待确认。

当前建议：

- 若持久化 store 已实现 `save_run(run)`，应评估同步签名，否则抽象不一致。
- 但不能引入数据库行为变更；只改接口兼容与测试。

### D5. PR-3 是否引入真实 RunIdentity generator？

推荐：**否**。

理由：

- PR-3 只迁移 RunStore 接受 external identity 的能力。
- `RunIdentity.create()` / run_id 格式选择属于 runtime-kernel identity contract，可在后续 Kernel.create Gate 冻结。
- PR-3 测试只需要固定 external run_id fixture，避免把 identity format 与 storage migration 绑定。

### D6. RunStore 是否允许保留 `generate_run_id` 形式的公开 API？

推荐：**否**。

理由：

- 公开 `RunStore.generate_run_id()` 会把 identity ownership 暗中留在 storage。
- migration window 允许内部 fallback mint id，但必须是 deprecated compatibility branch，不应暴露为新的 public API。
