# ADR 0005：AI 评审协作流程与分支模型

状态：已采纳
日期：2026-06-22

## 背景

ShanHai 采用「Trae/Claude/GPT 编码 → Git → ChatGPT 架构评审」的协作模式。需要把评审链路压到最短，同时让任意 AI 能快速、稳定地理解项目并定位评审范围。直接读取新建/未索引的 GitHub 仓库可能失败，需要兜底读取方式。此外，频繁直推 `main` 不利于评审与回滚。

## 决定

1. 新增固定入口文件：
   - `docs/PROJECT_STATE.md`：项目实时状态（版本、进度、不变量、下一步）。
   - `REVIEW.md`：评审入口，定义读取顺序、仓库 Raw/Diff 兜底地址、关注点与产出格式。
   - 与既有 `README.md` / `AGENTS.md` / `docs/架构决策记录/` 构成 AI 理解项目的固定入口集。
2. 采用分支模型 `main ← develop ← feature/*`：开发在 feature 分支，评审针对 feature 分支或指定 commit，通过后合入 develop，阶段稳定后合入 main。
3. 评审触发：提供「仓库链接 + commit SHA + 本次目标」，或简短 `Review`。
4. 仓库读取失败时的兜底：优先 Raw 文件与 commit diff，不走打包 zip。

## 原因

- 固定入口文件让评审者无需扫全仓即可对齐上下文，链路短且稳定。
- 分支模型保证 `main` 稳定、评审范围明确、变更可回滚。
- Raw/Diff 兜底解决新仓库无法直接索引的问题。

## 影响

- 每完成阶段/重要模块需同步更新 `docs/PROJECT_STATE.md`（纳入「完成模块后更新文档」约定）。
- `main` 不再接收开发性直推；后续 feature 分支按需创建。
- 评审产出遵循 REVIEW.md 约定格式，便于「通过/修改」决策。
