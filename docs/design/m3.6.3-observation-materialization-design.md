## 1. M1 — Materializer 职责边界（冻结，Blocking 1 已拆解）

核心组件 **`ObservationMaterializer`**：把一条 `ObservationDraft`（有语义、无身份）**加上一个已算好的 `ObservationIdentity`**，确定性地组装成一条 `Observation`（spine 读侧 DTO）。

**关键重构（Blocking 1）**：Materializer **不再自己拼 `logical_key` / 算 `content_hash`**——那属于 **Identity Policy**，已独立为 `IdentityStrategy`（§2）。Materializer 只做**纯组装**（Draft + Identity → Observation），**不含任何 fact_type 分支、不含任何 key 拼接规则**。这样以后 Financial / Quote / Announcement / Industry / Macro 的身份规则变化，只动 `IdentityStrategy` / `EffectiveScopeStrategy`，**Materializer 永远不用动**。

**两者都是纯函数、无 I/O**——把「算身份」（纯、可确定性单测）与「写 spine」（有 I/O）彻底分开，Identity 算法因此可脱离数据库、网络、provider 独立冻结与测试。

```
✅ MUST（Materializer 负责）                 ❌ MUST NOT（Materializer 禁止）
────────────────────────────────         ──────────────────────────────────────────
调 IdentityStrategy 取 Identity（§2）        自己拼 logical_key / 算 content_hash（归 IdentityStrategy）
从 draft 搬运语义字段到 Observation          出现任何 fact_type 分支 / effective_scope 规则（归 §2.4）
把 Identity + draft 组装成 Observation       发 HTTP / 取数（那是 Provider/Adapter，M3.6.4+）
设 confidence = spine 默认（1.0，非身份）    import / 调用任何 Provider
决定 Provenance 归属（§4）                  调 LLM / model-router / reasoning（Boundary 3）
                                          产 Belief / KnowledgeObject / Revision（Boundary 2）
                                          访问 Evolution / import market-intelligence（Boundary 4）
                                          直接写 SQLite / 碰 schema（写走 §5 的 writer 端口）
                                          解释 effective_at / 做 as_of / latest 投影（那是 Evolution）
```

- **输入**：一条（或一批）`ObservationDraft` + 一个注入的 `IdentityStrategy`。**输出**：一条（或一批）`Observation` 读侧 DTO。**无副作用**。
- **`observation_id` 不由 Materializer 造**：读侧 `Observation` DTO（[observation_reader.py](../../services/market-data/shanhai_market_data/ports/observation_reader.py#L51-L61)）本就**没有** `observation_id` 字段——它是 SQLite 行的存储态 surrogate，由 spine 在写入时分配（[sqlite_repository.py](../../services/market-data/shanhai_market_data/sqlite_repository.py#L467-L501)）。Materializer 只产 `(logical_key, content_hash, …)`，`observation_id` 仍归存储层——**呼应「不改 spine」**。

> 契约草图（**非实现**）：
> ```python
> # observation_materializer.py（M3.6.3 才写）
> class ObservationMaterializer:
>     """ObservationDraft + ObservationIdentity → Observation。纯组装，无 I/O，无 key 规则。"""
>     def __init__(self, identity: IdentityStrategy) -> None: ...
>     def materialize(self, draft: ObservationDraft) -> Observation:
>         ident = self._identity.identify(draft)          # (logical_key, content_hash) 全在 §2
>         return Observation(logical_key=ident.logical_key, content_hash=ident.content_hash, ...draft 字段...)
>     def materialize_many(self, drafts: tuple[ObservationDraft, ...]) -> tuple[Observation, ...]: ...
> ```

## 2. M2 — Identity 模型（冻结，一次定死，改则历史全废）

> **为什么必须现在冻结**：identity 是 append-only spine 的主键语义。今天 `subject+predicate`、明天 `subject+predicate+source`，会让**已落库的历史全部对不上号**（同一条逻辑事实被算成两条时间线，回放失真）。所以本节把公式定死，未来只能 **additive 扩展**（新 fact_type 声明自己的 scope 规则），**不能改既有公式**。

Identity 模型 = 三个可独立冻结的组件（Blocking 1/2 + Non-blocking 2）：

```
EffectiveScopeStrategy   §2.4  per fact_type：draft → effective_scope（唯一随 fact_type 扩展处）
IdentityStrategy         §2.2/§2.3  logical_key 模板 + content_hash 公式（跨 fact_type 稳定）
CanonicalSerializer      §2.5  跨语言确定性序列化（Python/Rust 同 bytes → 同 hash）
```

### 2.1 两个身份字段的语义分工（冻结）

| 字段 | 语义 | 回答的问题 |
|--|--|--|
| `logical_key` | **timeline 身份**（稳定） | 「这是哪一条逻辑事实的时间线？」——同一条事实的历次观测/修订共享它 |
| `content_hash` | **取值指纹** | 「这条时间线在这次观测里的值是什么？」——值变则 hash 变，产生新的一行（一次修订） |

spine 幂等键 = `(logical_key, content_hash)`（既有，不改）。二者由 `IdentityStrategy` 产出，封装为一个不可变 `ObservationIdentity(logical_key, content_hash)` 交给 Materializer。

> 契约草图（**非实现**）：
> ```python
> @runtime_checkable
> class IdentityStrategy(Protocol):
>     """draft → (logical_key, content_hash)。唯一定身份处；不含 I/O。
>     依赖注入 EffectiveScopeStrategy（§2.4）+ CanonicalSerializer（§2.5）。"""
>     def identify(self, draft: ObservationDraft) -> ObservationIdentity: ...
> ```

### 2.2 `logical_key`（冻结模板）

```
logical_key = "{entity_type}:{entity_id}|{fact_type}|{predicate}|{effective_scope}"
```

- `entity_type` / `entity_id`：来自 `draft.subject`（`label` **不参与**身份，仅展示）。
- `fact_type`：`draft.fact_type.value`。
- `predicate`：`draft.predicate or ""`（细分语义键，如 `valuation.pe_ttm`）。
- `effective_scope`：由 **`EffectiveScopeStrategy`**（§2.4）按 fact_type 算出的生效范围键。**latest-only 指标**（无周期性，如实时估值/画像）此段为 `""`。

**这是一个模板，不是 fact_type 分支**：`IdentityStrategy` 只按上式填空，**不 case fact_type**；随 fact_type 变化的只有 `effective_scope` 的取值规则，而那被隔离在 `EffectiveScopeStrategy`。

**设计理由（对齐 legacy 惯例，而非 sketch）**：legacy spine 早已把生效范围嵌进 `logical_key`（`fact:{sid}:quote:{trade_date}`、`financial:{sid}:{report_period}:{metric}`，见 [fact_mapper.py](../../services/market-data/shanhai_market_data/fact_mapper.py#L119-L181)）。M3.6 design §3/§5 的 sketch（`security:600519|financial|valuation.pe_ttm`，**不含周期**）与之不一致——本文采纳 **legacy 已验证的惯例**（把 `effective_scope` 纳入 `logical_key`），使每个交易日的行情、每期的财报各成一条时间线，「修正/重述」才是同一时间线上的新 `content_hash`。sketch 的旧格式登记入 §8 Reconcile R5。

### 2.3 `content_hash`（冻结公式）

```
content_hash = "sha256:" + sha256( CanonicalSerializer.dumps([ logical_key, object_value, predicate ]) )
```

- 沿用既有 [`_content_hash`](../../services/market-data/shanhai_market_data/sqlite_repository.py#L1176-L1185) 的编码风格（`sha256` + 稳定 JSON），并对齐既有 `"sha256:"` 前缀惯例（见 [test_public_providers.py](../../tests/market_data/test_public_providers.py#L127) 断言 `hash.startswith("sha256:")`）。序列化统一走 `CanonicalSerializer`（§2.5），不裸调 `json.dumps`。
- **纳入**：`logical_key`（自带 subject/fact_type/predicate/effective_scope）+ `object_value` + `predicate`。
- **刻意排除**（此即用户契约测试的规格，§6）：

| 排除字段 | 理由 |
|--|--|
| `captured_at` | knowledge time，不是取值本身；同值不同抓取时刻 → identity 不变（§6 用例④） |
| `source_ref` / `provider` | 数据源不参与身份；同值不同源 → identity 不变（§6 用例③），「换源不改身份」 |
| `confidence` | 信念，属认知层，不属观测身份（M3.6.0 §1.3） |
| `occurred_at` / `published_at` 原始时间戳 | 生效时间只以**粗粒度** `effective_scope` 进 `logical_key`；原始时间戳不进 hash，避免两源同一交易日时间戳差几秒就裂成两条身份 |

- **确定性**：同一 draft 恒得同一 `(logical_key, content_hash)`（§6 用例①）。

### 2.4 `EffectiveScopeStrategy`（冻结，Blocking 2，每个 fact_type 写清）

`effective_scope` **不是一句「归一到自然粒度」**——它是 per-fact_type 的**具名键**，由独立组件 `EffectiveScopeStrategy` 产出，避免以后每个 Provider 自己理解。冻结如下映射（新增 fact_type 只在此表 additive 加一行）：

| fact_type | effective_scope = | 取自 draft 的 | 归一规则 | 例 |
|--|--|--|--|--|
| `QUOTE` | **`trade_date`** | `occurred_at` | 归一到交易日（date，`YYYY-MM-DD`） | `2026-07-01` |
| `FINANCIAL` | **`report_period`** | `occurred_at`（报告期末） | 归一到报告期末日（date） | `2026-03-31` |
| `ANNOUNCEMENT` | **`announcement_id`** | `source_ref.external_id` | 原样取披露件唯一号（非时间） | `600519_2026-063` |
| `INDUSTRY` | **`industry_version`** | `occurred_at`（分类版本生效日） | 归一到版本生效日（date） | `2026-01-01` |
| `POLICY` / `NEWS` / macro 类 | **`publish_period`** | `published_at`（优先）/ `occurred_at` | 归一到发布日（date） | `2026-07-01` |
| `PROFILE` | `""`（latest-only，画像属最新态，无周期） | — | 空段 | `` |
| `ANOMALY` / `CAPITAL_FLOW` / `SHAREHOLDER` | **`event_date`** | `occurred_at` | 归一到事件日（date） | `2026-07-01` |

> 契约草图（**非实现**）：
> ```python
> @runtime_checkable
> class EffectiveScopeStrategy(Protocol):
>     """fact_type → effective_scope 具名键。唯一随 fact_type 扩展的地方。"""
>     def scope_of(self, draft: ObservationDraft) -> str: ...
> ```

- **每个键都是具名语义，不是匿名 `effective_scope`**：`QUOTE→trade_date`、`FINANCIAL→report_period`、`ANNOUNCEMENT→announcement_id`（用户点名三例，其余按同法冻结）。以后所有 FactType 照此表扩。
- **`ANNOUNCEMENT` 用 `announcement_id` 而非日期**：披露件天然有唯一号，同一天可多份公告；用 id 才能各成时间线（对齐 legacy `announcement:{company_id}:{announcement_id}`，见 [fact_mapper.py](../../services/market-data/shanhai_market_data/fact_mapper.py#L119-L181)）。
- **粗粒度是刻意的**（时间类 scope）：保证「换源不改身份」——EastMoney 与 akshare 对同一交易日的 close，即使 `occurred_at` 时间戳差几秒，也落到同一 `trade_date` → 同一 `logical_key`。
- **扩展只增不改**：新增 fact_type 或需更细粒度（如 intraday tick）→ 为**新类型/新键**声明规则，**既有类型的键冻结不动**（改既有 = 历史迁移，代价极高）。
- **源表示归一 vs 语义身份归一分工**：provider `mapper`（M3.6.4+）**必须**把各源时间/编号表示映射成语义一致的 `occurred_at/published_at/external_id`（源表示层归一是 mapper 职责）；`EffectiveScopeStrategy` 再把它们归一成 scope 键（语义身份归一职责）——两者分工登记，避免「换源改身份」。

### 2.5 `CanonicalSerializer`（冻结，Non-blocking 2，跨语言确定性）

`content_hash` 的稳定性取决于**序列化字节完全一致**。今天 Python `json.dumps`、明天若 Rust 侧也算 hash，稍有差异（浮点、时间、None、布尔、键序）就会 hash 不一致，历史全废。故把序列化抽成一个冻结组件 `CanonicalSerializer`，明确各类型的规范表示：

| 类型 | 规范表示（冻结） |
|--|--|
| dict / 对象 | 键按 UTF-8 字典序排序（`sort_keys=True`） |
| `str` | 原样（`ensure_ascii=False`，UTF-8） |
| `None` | JSON `null` |
| `bool` | JSON `true` / `false`（**不**与 0/1 混用） |
| `int` | 十进制整数原样 |
| `Decimal` / 金额 | **转字符串**按定点表示，不走 float（避免二进制浮点误差）；`object_value` 本就是 `str`，此规则为未来数值入 hash 预留 |
| `datetime` / `date` | ISO-8601 字符串（`date` = `YYYY-MM-DD`）；进 hash 的时间只有粗粒度 scope，原始时间戳已排除（§2.3） |
| 序列 | 保留顺序（顺序即语义） |

- 首版落地 = 既有 `json.dumps(payload, ensure_ascii=False, sort_keys=True, default=str)` 的**受约束封装**（沿用既有 `_content_hash` 风格），但**收敛到一个函数/类**，供 `IdentityStrategy` 独占调用——未来跨语言实现照此规范逐字段对齐即可。
- **Boundary**：`CanonicalSerializer` 是纯函数、无 I/O、不认识 fact_type，只认识「值 → 规范 bytes」。

## 3. M3 — Dedup 策略（冻结）

复用既有 spine 语义，**不新增策略**：

```
新 draft ─▶ materialize ─▶ (logical_key, content_hash)
                                    │
              ┌─────────────────────┴─────────────────────┐
       identity 已存在                              identity 不存在
              │                                            │
        ignore（幂等丢弃）                        append 新行（append-only）
     不刷新 captured_at / 不覆盖 / 不改历史
```

- **same hash → ignore**：完全相同的 `(logical_key, content_hash)` 幂等丢弃（既有 [`record`](../../services/market-data/shanhai_market_data/ports/in_memory_observation_reader.py#L51-L57) 行为，`self._seen` 去重）。
- **different hash（值变）→ new revision**：同 `logical_key`、新 `content_hash` = 追加一条新行，作为该时间线上的一次修订。历史行**永不被覆盖/删除**（append-only 是 bitemporal 回放的前提）。
- **明确不做**：❌ `refresh captured_at`（spine append-only，从不 mutate 已有行）；❌ 覆盖式 upsert；❌ 版本号自增（`version` 是认知层概念，不进观测身份）。
- **captured_at 归属**：同 identity 重复到达时，**保留首见行的 `captured_at`**（后到的幂等丢弃，其 `captured_at` 不采纳）——即 **first-seen wins**，与既有 `record` 行为一致。

> **Dedup ≠ Conflict Resolution（冻结边界，Non-blocking 3）**：Dedup **只处理「完全一样」**——即 `(logical_key, content_hash)` 逐字节相同者幂等丢弃。它**不判断**「哪个值对/新/更可信」，也**不合并**两个不同的取值。
>
> | | Dedup（本层，Materializer/spine） | Conflict Resolution（Evolution 层） |
> |--|--|--|
> | 处理对象 | 完全相同的 identity | 同 `logical_key`、**不同** `content_hash`（值有分歧） |
> | 动作 | 幂等丢弃 | 取舍 / 加权 / 置信度调和 / latest 投影 |
> | 归属 | M3.6.3 | ADR 0020 Knowledge Evolution |
>
> 值不同的两条观测在本层**都会各自 append 成行**（都是"系统曾观测到的事实"，append-only 全保留）；**谁是当前真相**由 Evolution 的 `KnowledgeResolver.resolve_at` 决定，**不在 Materializer**。这句话以后会救命——防止有人把冲突调和逻辑塞进 Materializer。

## 4. M4 — Provenance Merge（冻结）

问题：同一 `Observation`（同 `logical_key` + 同 `content_hash`，因 §2.3 排除了 source）来自 **EastMoney + CNInfo 两个源**，SourceRef 怎么办？

**裁决：first-writer-wins，单一 SourceRef，不 merge、不存多个。**

- 因为 `content_hash` 不含 source（§2.3），两源同值 → 同 identity → 第二条**幂等丢弃**（§3）→ 天然**保留首个写入者的 `SourceRef`**。
- **不做多源 provenance 列表**：那需要改 spine schema（新增 provenance 关系/多 SourceRef 存储）——违反 M5「不改 spine」。故 M3.6.3 **只保留单一来源指纹**。
- **代价（明确登记，非遗漏）**：本阶段**无法表达「两个源都印证了同一事实」**（cross-source corroboration / 可信度增强）。这是一个**未来项**，待多源可信度需求出现时，另开设计（可能引入 provenance 侧表或 `EvidenceRef` 聚合），**且不改既有 identity**。
- **与 confidence 的关系**：本阶段 `confidence` = spine 默认 `1.0`；「多源印证 → 提升 confidence」属认知层（Evolution）职责，**不在 Materializer**。

> **扩展点（冻结方向，Non-blocking 1）**：provenance 的演进路径**现在就钉死方向**，避免以后推翻 Observation——
>
> ```
> today:   Observation.source_ref : SourceRef          （单一，first-writer-wins）
>    ↓ future（不改 identity，仅 SourceRef 升级）
> future:  Observation.source_refs : tuple[SourceRef]  （多源印证；additive）
> ```
>
> 关键保证：因为 `content_hash` **不含 source**（§2.3），从「单 SourceRef」升到「SourceRef 列表」**不改变 `(logical_key, content_hash)`**——历史 identity 不动，只是同一条 Observation 的来源字段从 1 个变多个。届时是**字段扩展**（additive），**不是 Observation 重构**。M3.6.3 **不实现**该扩展，只保证公式对它前向兼容。

## 5. M5 — Spine 接缝（冻结）

**不改 spine。** Materializer 产 `Observation` 后，最终仍走既有 `record_many()`；M3.3 / M3.4 一行不动。

```
drafts ─▶ ObservationMaterializer.materialize_many ─▶ observations
                                                           │
                                                           ▼
                                              _ObservationWriter.record_many(observations)
                                                           │
                                                           ▼
                                        既有 append-only spine（InMemory / SQLite 两适配器已实现）
```

- **窄写端口 `_ObservationWriter`（新增，仅 M3.6.3）**：把 spine 写侧收敛成一个最小 Protocol，让写侧编排**依赖接口而非具体适配器**——

  > 契约草图（**非实现**）：
  > ```python
  > @runtime_checkable
  > class _ObservationWriter(Protocol):
  >     def record(self, observation: Observation) -> None: ...
  >     def record_many(self, observations: tuple[Observation, ...]) -> None: ...
  > ```
  > 既有 `InMemoryObservationReadPort` / `SQLiteObservationReadPort` **已满足此签名**（[in_memory](../../services/market-data/shanhai_market_data/ports/in_memory_observation_reader.py#L51-L61) / [sqlite](../../services/market-data/shanhai_market_data/ports/sqlite_observation_reader.py#L99-L109)）——端口是 structural fit，**不需要改它们**。

- **薄编排 `ingest_drafts`（push 模型，冻结形态）**：

  > 契约草图（**非实现**）：
  > ```python
  > def ingest_drafts(
  >     drafts: tuple[ObservationDraft, ...], *, writer: _ObservationWriter,
  >     materializer: ObservationMaterializer,   # 内部持 IdentityStrategy（§2）
  > ) -> IngestReport:
  >     observations = materializer.materialize_many(drafts)
  >     writer.record_many(observations)   # 幂等由 spine 保证（§3）
  >     return IngestReport(ingested=len(observations), ...)
  > ```
  - **组装分层**：`materializer = ObservationMaterializer(IdentityStrategy(EffectiveScopeStrategy(), CanonicalSerializer()))`——身份策略经依赖注入组装（§1/§2），`ingest_drafts` 本身**不认识**任何 key 规则。
  - **输入是 drafts，不是 provider**：刻意**不**在 M3.6.3 依赖 `provider.fetch_observations`——M3.6.2 的组合根 factory 尚是 pending（`NotImplementedError`），且真实 adapter 属 M3.6.4。用**手工构造的 draft / mock draft** 即可端到端验证 `draft → Observation → spine → 读回`，不被未实现的 adapter 阻塞。
  - **provider-pull 编排**（`provider.fetch → ingest_drafts`）是一行组合，**留待 M3.6.4** 有真实/mock provider 时接上。
- **读回验证**：写入后经既有 `ObservationReadPort.query(subject, knowledge_at=...)` 读回，证明 `draft → Observation → spine → ReadPort → （Evolution/Context 零改动）` 链路成立。

## 6. M6 — 契约测试规格（冻结，实现期落地）

**Identity 算法可纯函数单测（无 I/O）**，这是把它单独冻结的最大收益。至少覆盖用户点名的 5 条 + 7 条补充（含 rev 2 新增 ⑪⑫⑬）：

| # | 用例 | 断言 |
|--|--|--|
| ① | same draft → same hash | 同一 draft 两次 materialize，`(logical_key, content_hash)` 逐字段相等（确定性） |
| ② | different draft（值不同）→ different hash | `object_value` 不同 → `content_hash` 不同 |
| ③ | source different, content same → identity same | 仅 `source_ref` 不同（provider/hash/trust 不同），`(logical_key, content_hash)` **相等**（换源不改身份） |
| ④ | captured_at different, content same → identity same | 仅 `captured_at` 不同，`(logical_key, content_hash)` **相等** |
| ⑤ | predicate changed → new Observation | `predicate` 不同 → `logical_key` 不同 → 新 Observation（新时间线） |
| ⑥ | idempotent re-ingest | 同一批 draft `ingest_drafts` 两次，spine 行数不增（`record_many` 幂等，§3） |
| ⑦ | value revision → new row same timeline | 同 subject/fact_type/predicate/effective_scope、`object_value` 变 → 同 `logical_key`、新 `content_hash`、spine +1 行（修订入历史） |
| ⑧ | provenance first-writer-wins | 两源同值先后 ingest，读回该 Observation 的 `source_ref` = **首个**源；spine 不新增第二行（§4） |
| ⑨ | effective_scope granularity | 同 subject 同 predicate、不同交易日 → 不同 `logical_key`（各成时间线）；同交易日不同秒级 `occurred_at` → **同** `logical_key`（粗粒度，§2.4） |
| ⑩ | boundary AST guard | `observation_materializer` / `ingest` 模块 **不 import** market-intelligence、不含 LLM/reasoning 符号、不产 Belief/Knowledge（对齐 [test_dependency_boundary.py](../../tests/market_data/test_dependency_boundary.py) 的 AST 守护风格；M3.6 design §7 Boundary 守护落地） |
| ⑪ | scope key per fact_type（Blocking 2） | `EffectiveScopeStrategy`：QUOTE 用 `trade_date`、FINANCIAL 用 `report_period`、ANNOUNCEMENT 用 `announcement_id`（=`source_ref.external_id`，非日期）、PROFILE 用 `""`——各按 §2.4 表产出预期 `logical_key` |
| ⑫ | canonical serializer 稳定（Non-blocking 2） | 键序打乱的等价 payload / `None` / `bool` 经 `CanonicalSerializer` 产出**同 bytes → 同 hash**；float 与等值 Decimal-string 的差异被规范表示消解（回归 hash 稳定性） |
| ⑬ | dedup ≠ conflict（Non-blocking 3） | 同 `logical_key`、**不同** `object_value`（值分歧）两条 ingest → spine **保留两行**（都 append，不合并/不取舍）；本层**不**判定谁对（冲突调和留 Evolution，§3） |
| ⑭ | identity decoupled from materializer（Blocking 1） | 注入一个 stub `IdentityStrategy`（返回固定 identity）→ Materializer 输出的 `(logical_key, content_hash)` = stub 值，证明 Materializer **不含**自有 key 规则（策略可替换） |

- 测试范式对齐既有 `tests/market_data/`：`PYTHONPATH=services/market-data:. .venv/bin/python -m tests.market_data.test_x`；`from __future__ import annotations`；纯 `assert` + `print("[OK] …")` + `def main()`；无 pytest。
- 实现后跑**全量 market_data 回归**绿（不回归 M3.2/M3.3/M3.4）。
