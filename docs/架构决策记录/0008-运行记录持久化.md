# ADR 0008：运行记录持久化

状态：已采纳
日期：2026-06-22

## 背景

Phase 1 的 `AgentRunner` 已把每次运行结构化为 `RunResult`（`agent` / `status` / `output` / `steps[]` / `error`），其中 `steps` 逐条记录 think / act / observe（含工具调用与结果）。但这些记录目前**只存在于内存**，运行结束即丢失。

ADR 0006 已明确：「持久化运行记录（落库）留待数据层接入阶段，另开 ADR」。本 ADR 即为该决策。

落库的价值（对应「知识优先」原则）：
- 让「运行过程」成为一等数据，支撑 `evaluation` 服务做回归与质量度量；
- 为多 Agent 研究系统提供可追溯、可复盘的执行轨迹；
- 为后续「从历史经验中学习」沉淀原始素材。

约束（AGENTS.md 铁律）：
- Agent 禁止直接访问数据库，调用链为 `Agent → Tool → Service → Database`（§4）。
- 模块独立：`agent-runtime` 是通用基础设施，不应直接依赖具体数据库驱动。

## 待决问题（评审重点）

1. **持久化的触发位置**：由 `AgentRunner` 经注入的 `RunStore` 落库（运行时横切关注点，类比日志），还是要求 Agent 经 Tool 主动写库？
2. **存储后端的归属**：DB 实现放在哪个模块，才能既落库又不让 `agent-runtime` 依赖数据库驱动？
3. **Step 的存储形态**：独立 `agent_steps` 表，还是 `agent_runs` 表内 JSONB 列？
4. **失败语义**：持久化失败是否应让整次运行判失败？
5. **同步还是异步**：本阶段同步落库即可，还是引入队列？

## 决定（建议方案，待确认）

1. **定义存储抽象 `RunStore`（放 `agent-runtime`）。**
   - 接口：`save_run(run: RunResult) -> str`（返回 run_id）、`get_run(run_id) -> RunRecord | None`、`list_runs(agent: str | None, limit: int) -> list[RunRecord]`。
   - `agent-runtime` 只依赖该抽象，不依赖任何数据库驱动，保持模块独立。

2. **默认实现 `InMemoryRunStore`（放 `agent-runtime`）。**
   - 零外部依赖，用于测试与本机无 DB 场景；当前 `RunResult` 行为不变（默认不注入即不落库）。

3. **DB 实现放独立持久化层，不污染 `agent-runtime`。**
   - 新增 `services/persistence` 实现 `PostgresRunStore`，DB 驱动（psycopg）惰性导入，作为可选依赖 `[postgres]`；`agent-runtime` 仅在装配时按需注入。
   - 依赖方向：`persistence → agent-runtime`（抽象），单向。

4. **触发位置：`AgentRunner` 经注入的 `RunStore` 在运行结束后落库（best-effort）。**
   - 理由：运行记录是**运行时自身的可观测产物**（类似 trace/log），不是 Agent 的业务能力，不应反向要求每个 Agent 经 Tool 写库；Agent 仍完全不感知存储。
   - 默认不注入 store 时不落库，零行为变化；注入后由 runner 调用 `store.save_run(result)`。

5. **失败语义：持久化失败不改变运行结果（best-effort），仅记录错误。**
   - 落库异常被捕获并记录，不污染 `RunResult.status`；运行成功与否只取决于执行本身。

6. **数据模型（建议）：**
   - `agent_runs`：`id`(uuid) / `agent` / `status` / `output`(jsonb) / `error` / `created_at`。
   - `agent_steps`：`id` / `run_id`(fk) / `index` / `type` / `content` / `tool` / `tool_args`(jsonb) / `tool_result`(jsonb)。
   - 与 Wiki 表一致，由各模块 Schema 驱动建表（init SQL 暂留占位，建表脚本随持久化层落地）。
   - 备选：steps 直接以 JSONB 存于 `agent_runs.steps`，省一张表但牺牲按步查询能力。

7. **本阶段同步落库**，不引入队列；异步/批量留待吞吐成为瓶颈时另议。

## 原因

- 用抽象 `RunStore` + 依赖注入，既满足「落库」又守住「`agent-runtime` 不依赖 DB、模块独立」。
- 把 DB 实现隔离到独立持久化层，符合「Service 边界」精神：Agent 与 runtime 不感知具体存储。
- best-effort 落库保证可观测性不反噬主流程可靠性。
- 默认 `InMemoryRunStore` 让测试与无 DB 环境零成本，符合当前本机无 Docker 的现实。

## 影响

- `agent-runtime` 新增 `store.py`（`RunStore` 抽象 + `InMemoryRunStore` + `RunRecord`），`AgentRunner` 增加可选 `store` 注入与 best-effort 落库；`BaseAgent` 支持透传 `store`。
- 新增 `services/persistence`（`PostgresRunStore` + `init_schema`），建表 SQL 落在 `infrastructure/database/init/002_agent_runs.sql`；本机无 Docker，DB 实现仅静态检查 + 以 `InMemoryRunStore` 跑通契约测试。
- `evaluation` 服务后续可基于 `RunStore.list_runs` 读取轨迹做度量。
- 不改变 Agent 编写方式；不注入 store 时行为与现状完全一致。

## 备选方案（已考虑）

- **Agent 经 Tool 写库**：契合「Agent→Tool→Service」字面，但会把「记录运行」变成每个 Agent 的显式职责，侵入业务、易遗漏，且运行记录本质是 runtime 横切产物，故不采纳。
- **直接在 `agent-runtime` 写 Postgres**：简单，但让基础设施模块依赖 DB 驱动，破坏模块独立，不采纳。
- **仅 JSONB 单表**：实现最简，但失去按 step/tool 维度查询与统计能力，作为备选保留。
