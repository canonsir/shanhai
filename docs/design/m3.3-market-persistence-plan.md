## 1. 现状 Review（开工前的事实基线）

### 1.1 store 的消费面（领域层今天依赖什么）

`InMemoryMarketKnowledgeStore`（[store.py](../../services/market-data/shanhai_market_data/store.py)）对外暴露 **8 个方法**，分两类：

**写入（ingestion 调用）** —— `acquisition.py` / `sync.py` 使用：

- `upsert_company_bundle(*, company, listed_entity, security, listing, industry?, facts, financial_facts, announcement_facts)`
- `upsert_market_facts(company_id, facts)` / `upsert_financial_facts(...)` / `upsert_announcement_facts(...)`
- `upsert_quote(quote)`

**读取（read model 调用）** —— `api.py` (`CompanyIntelligenceAPI`) 使用：

- `get_company_intelligence_by_ts_code(ts_code) -> CompanyIntelligence | None`
- `get_company_timeline(ts_code, *, time_basis, latest_first) -> tuple[CompanyTimelineEvent, ...]`
- `list_company_intelligence(limit) -> tuple[CompanyIntelligence, ...]`
- `search_company(text, limit) -> tuple[CompanyIntelligence, ...]`

> 这 8 个方法就是 **repository contract 的事实草案**——已被 `api.py`、`acquisition.py`、
> `sync.py` 真实依赖。M3.3 的 Repository 抽象必须无损覆盖它们，业务层零改动。

### 1.2 已存在但漏抽象的 Postgres 实现（要替换的反面教材）

[postgres_store.py](../../services/market-data/shanhai_market_data/postgres_store.py) 目前：

- **继承** `InMemoryMarketKnowledgeStore`（`class PostgresMarketKnowledgeStore(InMemoryMarketKnowledgeStore)`），用「内存装配 + 落一张 JSONB 表」的影子写法；
- 只有 **一张表** `market_company_intelligence(ts_code, company_id, company_name, payload JSONB, updated_at)`——把整个 read model 序列化成 blob；
- 因此**无法** query history / compare changes / 按 fact 维度检索——正是用户指出的缺口。

```
问题：Postgres = InMemory 的子类 + 一坨 JSONB
      → 存储实现污染领域层
      → 没有真正的 fact / entity 表
      → replay / 历史查询 / 多源比对都做不到
```

**结论**：M3.3 不是「给 InMemoryStore 接个 DB」，而是**先抽出 Repository 边界，再让存储实现退到边界后面**；现有 `PostgresMarketKnowledgeStore` 的继承式写法在 M3.3 被规范化的 PostgresRepository 取代（保留 commit 历史，不静默删能力）。

### 1.3 codebase 已有可直接镜像的 Repository 范式（关键 prior art）

`services/persistence`（ADR [0008](../架构决策记录/0008-运行记录持久化.md) / [0009](../架构决策记录/0009-Local-first持久化.md)）已经做过一模一样的事，**M3.3 直接沿用同一范式，不另起炉灶**：

| RunStore 范式（已落地）                     | M3.3 对应物（本计划）                          |
| ------------------------------------------- | ---------------------------------------------- |
| `RunStore` 抽象（ABC，放 agent-runtime）    | `MarketKnowledgeRepository` 抽象               |
| `InMemoryRunStore`（零依赖，测试/无 DB）    | `InMemoryMarketKnowledgeRepository`            |
| `SqliteRunStore`（local-first 默认，stdlib）| `SqliteMarketKnowledgeRepository`（local-first）|
| `PostgresRunStore`（增强后端，psycopg 惰性）| `PostgresMarketKnowledgeRepository`（增强）    |
| `default_run_store()` 按环境变量装配        | `default_market_repository()` 按环境变量装配   |
| 「DB 是增强、非开发前置」                   | 同方针（见 §4 Migration / 开放问题 A）         |

> 这是本计划最重要的一致性约束：M3.3 不发明新的持久化哲学，**复用 ShanHai 已批准的
> local-first 持久化 ADR（0008/0009）**——同一抽象 + 多后端 + 装配工厂 + DB 驱动惰性导入。

## 2. Repository Boundary（第一优先级）

### 2.1 原则

> 不要直接把 InMemoryStore 换 PostgreSQL，中间**必须**加 Repository。

否则 Postgres / SQLite / DuckDB / Vector DB 都会污染业务。领域层只认 `MarketKnowledgeRepository` 抽象，**完全不感知**底层是内存、SQLite 还是 Postgres。

### 2.2 目标目录结构

`services/market-data/shanhai_market_data/` 演化为：

```
shanhai_market_data/
├── domain/                       # 领域模型（现 models.py 拆入或保留 re-export）
│   └── models.py                 # Company / Security / *Fact / CompanyIntelligence ...
│
├── repository/                   # ★ 边界：抽象 + 读写契约（不依赖任何 DB 驱动）
│   ├── __init__.py
│   └── market_knowledge_repository.py   # MarketKnowledgeRepository (ABC)
│
├── memory/
│   └── in_memory_repository.py   # InMemoryMarketKnowledgeRepository（由现 store.py 演化）
│
├── infra/
│   ├── sqlite/
│   │   └── sqlite_repository.py  # SqliteMarketKnowledgeRepository（local-first 默认）
│   └── postgres/
│       └── postgres_repository.py # PostgresMarketKnowledgeRepository（增强后端）
│
├── factory.py                    # default_market_repository() 按 env 装配
├── acquisition.py                # 写：依赖 repository 抽象（改注入类型）
├── sync.py                       # 写：依赖 repository 抽象（改注入类型）
├── api.py                        # 读：依赖 repository 抽象（改注入类型）
├── timeline.py                   # 纯函数 read model，不动
├── provider.py / providers/      # 采集边界，不动
├── mapper.py / fact_mapper.py    # 映射层，不动
├── resolver.py / registry.py / identity.py   # 身份层，不动
```

> 命名收敛：用户原文给的 `market_fact_repository.py / company_repository.py /
> timeline_repository.py` 是「按聚合拆多个 repo」的写法。**本计划建议第一版先用单一
> `MarketKnowledgeRepository`**（与现 store 8 方法 1:1，迁移零风险、与 RunStore 范式一致），
> 待表数量和查询面长大后再按聚合拆分。→ 见**开放问题 B**，请 Review 拍板。

### 2.3 Repository 契约（草案，1:1 覆盖现 store 消费面）

```python
class MarketKnowledgeRepository(ABC):
    # --- write (ingestion) ---
    def upsert_company_bundle(self, *, company, listed_entity, security,
                              listing, industry=None, facts=(),
                              financial_facts=(), announcement_facts=()) -> None: ...
    def upsert_market_facts(self, company_id, facts) -> None: ...
    def upsert_financial_facts(self, company_id, facts) -> None: ...
    def upsert_announcement_facts(self, company_id, facts) -> None: ...
    def upsert_quote(self, quote) -> None: ...

    # --- read (read model) ---
    def get_company_intelligence_by_ts_code(self, ts_code) -> CompanyIntelligence | None: ...
    def get_company_timeline(self, ts_code, *, time_basis, latest_first) -> tuple[CompanyTimelineEvent, ...]: ...
    def list_company_intelligence(self, limit=50) -> tuple[CompanyIntelligence, ...]: ...
    def search_company(self, text, limit=50) -> tuple[CompanyIntelligence, ...]: ...
```

依赖方向（单向，与 ADR 0008 同精神）：

```
infra/{sqlite,postgres}  ──┐
memory/in_memory_repo    ──┼──►  repository/ (抽象)  ◄──  api / acquisition / sync
                           │                                （业务只依赖抽象）
domain/models  ◄───────────┘
```

`repository/` 与 `domain/` **不导入任何 DB 驱动**；`psycopg` 仅在 `infra/postgres/` 惰性导入（无 DB 环境仍可 import + 静态检查 + 用 memory/sqlite 跑契约测试）。

## 3. 数据库 Schema（第二优先级）

### 3.1 优先级

按用户指示，**优先 Company / Entity**（Console 已依赖），再 fact 系列。但落库 schema 必须尊重 M2.5 已冻结的**四层身份模型**（不可塌缩）：

```
Company ──< ListedEntity ──< Security ──< Listing
                                   └─ ts_code 只是 Security 的属性/查找键，不是 identity
```

> 用户草案里的 `company / security` 两表是简化表达。实际落库要保留 ListedEntity、
> Listing 两层（否则 `CompanyIntelligence` read model 装不出来）。**ts_code 仍只是
> `security.ts_code` 一列**，主键一律用 surrogate `*_id`（`new_internal_id` 分配）。
> 见**开放问题 C**。

### 3.2 表清单（逻辑模型草案，SQLite 与 Postgres 共用，仅方言适配）

身份与实体（current truth）：

```
company
  company_id      PK (surrogate)
  name
  region
  aliases         json
  external_ids    json
  created_at / updated_at

listed_entity
  listed_entity_id PK
  company_id       FK -> company
  disclosure_name
  source_ref       json
  created_at / updated_at

security
  security_id      PK
  listed_entity_id FK -> listed_entity
  ts_code          (属性/查找键，建唯一索引，非 identity)
  symbol / name / exchange / security_type / currency
  created_at

listing
  listing_id       PK
  security_id      FK -> security
  exchange / board / listed_at / delisted_at / status

industry
  industry_id      PK
  taxonomy / name / code / level

company_industry           # company ↔ industry 关联（现 store 用 dict 维护）
  company_id FK / industry_id FK
```

事实系列（每条 fact 一行，**不塌缩成单超表**，对齐三个 fact family）：

```
market_fact
  fact_id          PK
  subject_id       (= subject_ref.entity_id, surrogate)
  predicate
  object_value
  object_type      (= object_ref 的类型或 fact attribute 标注)
  fact_type
  occurred_at / published_at / captured_at   ← 三时态永不塌缩
  confidence
  source_ref       json (含 provider/dataset/raw_snapshot_ref/version/hash)
  attributes       json
  schema_version

financial_fact
  fact_id          PK
  security_id      FK -> security
  report_period / report_type / metric_name
  metric_value / unit / currency / yoy / qoq / restated
  occurred_at / published_at / captured_at
  confidence / source_ref(json) / schema_version

announcement_fact
  fact_id          PK
  company_id       FK -> company
  title / announcement_type(category) / announcement_id
  published_at / occurred_at / captured_at / ann_date
  document_url / document_hash(content_hash)
  extracted_summary / confidence / source_ref(json) / schema_version

quote_snapshot
  quote_id         PK
  security_id      FK -> security
  trade_date
  open/high/low/close/previous_close/volume/amount
  source_ref       json
```

设计要点：

- **三时态**（`occurred_at` / `published_at` / `captured_at`）每张 fact 表都保留独立列，
  绝不合并——这是 timeline 三时间语义和未来 replay/compare 的基础。
- **source_ref 第一版整体存 JSON 列**（含 M3.2 已落的 provider/dataset/raw_snapshot_ref/
  version/hash）。独立 `source_ref` 物化表 + 不可变 raw snapshot 是 **M3.4 的事**（见 §5）。
- **timeline 不建表**：`CompanyTimelineEvent` 是 read model，落库后仍由 `timeline.py`
  纯函数从三张 fact 表投影。用户草案里的「Timeline Read Model / timeline_repository」
  指的是读路径，不是物理表。
- 主键全 surrogate；`ts_code` 仅 `security` 一列 + 唯一索引（identity 不塌缩铁律）。

## 4. Migration Strategy

### 4.1 建表方式（沿用 ADR 0008/0009：`init_schema()` 幂等内置）

- 每个 infra 后端内置 `SCHEMA_SQL` + 幂等 `init_schema()`（`CREATE TABLE IF NOT EXISTS`），
  构造时自建表——**无 Docker 首跑即可用**（SQLite 落盘文件）。
- 同步在 `infrastructure/database/init/003_market_knowledge.sql` 落一份 Postgres 建表脚本
  （与 `002_agent_runs.sql` 并列，供容器初始化）。
- 暂**不引入 Alembic / SQLAlchemy**（当前无 ORM、无迁移历史；RunStore 也没用）。schema
  演进先靠幂等 `IF NOT EXISTS` + 加列脚本；待表结构频繁变更再评估迁移框架。→ **开放问题 D**。

### 4.2 数据回填

- 无历史数据需迁移：现 InMemoryStore 重启即空，没有存量。
- 切换后用 `POST /market/acquisition/public/run`（或 startup seed）重新 ingest 600519.SH
  等目标，数据直接落到 SQLite/Postgres——这就是回填。

### 4.3 本机现实（关键约束）

> 本机无 Docker、无 psycopg（venv 未装）。

因此 **local-first 默认必须是 SQLite**（stdlib `sqlite3`，零依赖、可落盘、可查询），
Postgres 为增强后端——与 ADR 0009 完全一致。Step 3 的 PostgresRepository 在无 DB 环境
下只做静态检查 + 用 SQLite/InMemory 跑契约测试（同 PostgresRunStore 的做法）。

→ 这引出**开放问题 A**：用户在 M3.3 校准里写的是「Memory → PostgreSQL（current truth）」，
但 ShanHai 已批准的 ADR 0009 是「local-first SQLite 默认 + Postgres 增强」。本计划**建议
遵循 ADR 0009**：默认 SQLite 落盘（重启不丢、无需 Docker），Postgres 作为 `SHANHAI_MARKET_STORE=postgres`
的增强后端。请 Review 确认是否接受这个与既有 ADR 一致的口径。

## 6. M3.3 不做什么（明确边界）

- ❌ **Raw Snapshot / 不可变原始快照存储** → 留 **M3.4 Knowledge Provenance**

- ❌ **按聚合拆多 repository、ORM、迁移框架、Vector/Graph DB** → 第一版不做（见开放问题 B/D）。
- ❌ 触碰 RuntimeKernel / RuntimeContext / Experience / Console 契约。

## 10. Phase 1.5 — Persistence Contract Freeze（Review PASS，2026-06-30）

> 本节是 Phase 1.5 Review 的收敛结果。开放问题 A–E 已在 Phase 1 拍板（见 §8 + Phase 1
> 实现），Phase 1.5 进一步**冻结「市场知识如何存在」的存储语义**。此处冻结的是**契约**，
> 不是实现——Phase 2 必须在此契约下实现，不得偷改语义。

### 10.0 一个前置概念修正：Assertion vs Observation

ShanHai 不是行情存储，而是 **AI Native 中国资本市场认知系统**。因此 Knowledge Fact 内部
必须区分两层（当前 `MarketFact` 把两者混在一起，这是后续要拆清的认知债）：

```
Knowledge Fact
   ├── Assertion（事实断言：逻辑上「这条事实是什么」）
   └── Observation（事实观测：某来源在某时刻「看到的值」）
```

- **logical_fact_key** = Assertion 的身份（如 `security=600519 / metric=closing_price /
  trade_date=2026-06-30`，即现在的确定性 `fact_id`）。
- **observation** = 同一 Assertion 下的一次带时间戳与来源的取值。

> 因此文档中**不写「Fact append-only」**（会误导成「行情流水」），而写下述四句存储语义。

### 10.1 Repository 写语义（FREEZE）

```
Repository storage semantic:

  1. Observation is append-only.
  2. Fact identity is logical identity.        (logical_fact_key, 不再作为物理 PK)
  3. Observation version is immutable.          (每个 observation 写入后不可改)
  4. Duplicate observation is prevented by content hash.
```

```
logical_fact_key
   ├── observation v1   (captured_at=t1, hash=A)
   ├── observation v2   (captured_at=t2, hash=B)
   └── observation v3   (captured_at=t3, hash=C)
```

两个奠基性场景（都「保留全部观测、不 update 覆盖」）：

- **盘中股价**：`600519 / closing_price / 2026-06-30` 在 10:00=1500(hashA)、15:00=1510(hashB)
  → 两个 observation 并存。
- **财报修订（restatement）**：`贵州茅台 / 2025Q4 / revenue` 初次 170B(hashA)、年报修订 172B(hashB)
  → 两个都留，因为「市场认知发生过变化」，这对未来 AI 推理至关重要。

幂等规则：重抓返回**相同值** → content_hash 命中 → **不新增行**（防重跑脏数据）；值变化 →
hash 不同 → 新增一个 observation。→ 兼得 append-only（历史不灭）+ 幂等（重跑不脏）+
restatement-aware。

> **现状与差距（实证）**：今天 [fact_mapper.py](../../services/market-data/shanhai_market_data/fact_mapper.py)
> 的 `fact_id` 确定性、不含 captured_at；[store.py](../../services/market-data/shanhai_market_data/store.py)
> 按 `bucket[fact.fact_id] = fact` **覆盖写**——即当前是 upsert，不是上面冻结的语义。
> `MarketFact/FinancialFact/AnnouncementFact` 三类**模型已自带 `captured_at`**，无需加字段；
> 缺的是 `content_hash` 填充（现 `SourceRef.hash=None`）。这些都属 **Phase 2 实现**，此处只冻语义。

对 SQLite 主键的指导（Phase 2 落实，**此处不建表**）：

```
PRIMARY KEY  : observation_id                        (surrogate, append-only)
UNIQUE       : (logical_fact_key, content_hash)        ← 幂等去重
INDEX        : (logical_fact_key, captured_at)         ← timeline 取「最新观测」
```

### 10.2 Timeline Projection（FREEZE）

Timeline 的定义被收紧为 **Company Knowledge Evolution Projection**，不是事件流水/行情 tick 流：

```
Facts / Financial Facts / Announcement Facts
        │  build_company_timeline()  (纯函数，绝不建表)
        ▼
Timeline Projection
   默认：latest observation per logical fact
```

```
Timeline = knowledge evolution
NOT      = tick / event stream
```

未来若要「10:00 1500 → 15:00 1510」这类逐 tick 视图，**单独建设 Market Observation Stream**，
不污染 timeline。完整观测历史走独立查询（如未来的 `get_fact_observations(logical_fact_key)`），
与 timeline 分离。

### 10.3 Identity Model（FREEZE）

四层 `Company ──< ListedEntity ──< Security ──< Listing` 保持，全 surrogate 主键。

- **不变量**：`ts_code` 是 **Security 的属性/查找键，永远不是 identity**，永远 ≠ `company_id`。
- **`Security.external_ids`：Deferred（不进入 Phase 2）**。在真正出现多 vendor code（EastMoney /
  Tushare ts_code / Wind / Bloomberg）之前，不为未来做 schema 扩张。当前 `security` 字段
  `id / symbol / ts_code / market` 已足够。
- 表述澄清：命名空间外部码（`tushare:ts_code:600519.SH`）当前落在 `Company.external_ids`，
  当前**不存在 `Security.external_ids` 字段**——这是已知的、被有意推迟的扩展点，不是 bug。

### 10.4 SourceRef / RawSnapshot（FREEZE 方案 B）

```
Fact ──> SourceRef ──> raw_snapshot_ref   (保留 ref 命名，不改成 raw_snapshot_id)
```

```
SourceRef owns locator semantics.     (M3.3：Fact 只持有一个中立 locator)
RawSnapshot owns storage semantics.   (M3.4：表/对象/Parquet/Git object 等形态由它决定)
```

> 保留 `raw_snapshot_ref`（而非 `raw_snapshot_id`）是刻意的：M3.3 不应感知未来 RawSnapshot
> 的存储形态（可能是 Postgres 表 / S3 对象 / Parquet 文件 / Git object），`ref` 比 `id` 更
> 架构中立。Fact 不感知存储细节。RawSnapshot 物化留 **M3.4 Knowledge Provenance**。

### 10.5 冻结结果总表

| 项 | 冻结 |
| --- | --- |
| Repository | Observation append-only + content hash idempotency |
| Fact ID | logical identity，不再作为物理 PK |
| Observation | immutable |
| Timeline | projection，不建表 |
| Timeline 默认 | latest observation per logical fact |
| Identity | 四层模型保持 |
| ts_code | Security 属性（非 identity） |
| Security.external_ids | 延后（不进 Phase 2） |
| SourceRef | 保持 locator（`raw_snapshot_ref`） |
| RawSnapshot | M3.4 |

### 10.6 登记概念：Knowledge Temporal Model（登记，不实现）

ShanHai 已不是简单 CRUD 库。核心查询会是「**2025 年 3 月时，市场认为贵州茅台是什么状态？**」
——这是 `knowledge state at time T`，不是 `current company state`。

```
Fact
 └── Observation history
       └── captured_at
       └── source
```

§10.1 的 append-only + content_hash 设计**正是为支撑时点知识重建**。本概念**登记在案、本阶段不实现**，
作为 Phase 2 之后（及 M3.4 provenance）的设计指北。

### 10.7 Phase 2 范围（已批准进入）

范围 = **implement frozen Repository contract**（不是「create tables」优先，而是「实现已冻结契约」）。

执行顺序：

```
1. SQLite repository      (实现 §10.1–10.4 冻结契约)
2. schema                 (PK/UNIQUE/INDEX 按 §10.1)
3. migration / init       (幂等 init_schema()，沿用 ADR 0008/0009，不引 Alembic)
4. repository tests       (契约一致性：同一组用例对 InMemory/SQLite 跑通)
5. replace default backend (default_market_store() 装配切 SQLite 默认)
```

Phase 2 **暂不做**：❌ Postgres ❌ RawSnapshot ❌ Alembic ❌ Console ❌ Design System ❌ AI Runtime。
