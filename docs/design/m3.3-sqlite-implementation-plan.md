## 1. Repository 方法 → Option C schema 映射

### 1.0 先校正契约事实（不静默接受伪代码 / 错误计数）

你指令里的 `save_company() / save_fact() / save_financial_fact() / get_company() /
list_company_facts() / get_timeline()` 是**说明性伪代码**。真实冻结契约
（[domain/repository.py](../../services/market-data/shanhai_market_data/domain/repository.py)）是：

```
写（ingestion，5）：
  upsert_company_bundle(*, company, listed_entity, security, listing,
                        industry=None, facts=(), financial_facts=(), announcement_facts=())
  upsert_market_facts(company_id, facts)
  upsert_financial_facts(company_id, facts)
  upsert_announcement_facts(company_id, facts)
  upsert_quote(quote)

读（read model，4）：
  get_company_intelligence_by_ts_code(ts_code) -> CompanyIntelligence | None
  get_company_timeline(ts_code, *, time_basis=PUBLISHED_AT, latest_first=True) -> tuple[...]
  list_company_intelligence(limit=50) -> tuple[...]
  search_company(text, limit=50) -> tuple[...]
```

> ⚠️ **计数对账（doc hygiene，需你确认）**：前序冻结文档反复写「8 方法（5 写 4 读）」，但
> 5 写 + 4 读 = **9 个方法**（timeline 被算进 4 读）。Protocol 实际暴露 9 个。本文一律以这 9 个
> 真实签名为准；建议后续 doc-hygiene 把「8」更正为「9（5 写 + 4 读）」，**不影响本阶段实现**。

一致性测试将对照**这 9 个真实方法**做，绝不测 `save_fact/get_fact` 伪代码（schema §4/§6 已声明）。

### 1.1 写入链路映射（write → spine + detail）

```
Provider → Fact Mapper → MarketKnowledgeRepository → SQLiteRepository
                                                          │
                                                knowledge_observation (spine, append-only)
                                                          ├── financial_observation
                                                          ├── quote_observation
                                                          └── announcement_observation
```

| Repository 写方法 | 落库目标（Option C） | 语义 |
| --- | --- | --- |
| `upsert_company_bundle(...)` | `company / listed_entity / security / listing (/ industry / company_industry)` **current-truth upsert**，再把 `facts / financial_facts / announcement_facts` 委托给下面三个 fact 写方法 | 身份表是 current truth（覆盖写）；fact 走 append-only |
| `upsert_market_facts(company_id, facts)` | 每条 `MarketFact` → 1 行 `knowledge_observation`（spine），无 typed detail（profile/industry/quote-as-fact 的结构都在 spine 列） | append-only + content_hash 幂等 |
| `upsert_financial_facts(company_id, facts)` | 每条 `FinancialFact` → 1 行 `knowledge_observation`（spine 共性）+ 1 行 `financial_observation`（`metric_name/report_period/report_type/metric_value/unit/currency/yoy/qoq/restated`） | spine + detail 1:1 |
| `upsert_announcement_facts(company_id, facts)` | 每条 `AnnouncementFact` → spine + `announcement_observation`（`announcement_id/announcement_type/title/publish_date/document_url/document_hash/extracted_summary`） | spine + detail 1:1 |
| `upsert_quote(quote)` | 见 §1.3（quote 双写问题） | — |

**spine 列填充来源**（实证字段，非新设计）：
- `logical_key` ← 现 `fact_id`（如 `fact:{company_id}:profile:region` /
  `financial:{security_id}:{period}:{metric}` / `announcement:{company_id}:{ann_id}`，见
  [fact_mapper.py](../../services/market-data/shanhai_market_data/fact_mapper.py)）。
- `fact_type / subject_type(=subject_ref.entity_type) / subject_id(=subject_ref.entity_id) /
  predicate / object_value` ← 现有 fact 模型既有字段。
- `occurred_at / published_at / captured_at` ← 三类 fact 模型**已自带**（captured_at 有 default factory）。
- `content_hash` ← **Phase 2 需补填**（现 `SourceRef.hash=None`、`fact_id` 不含 captured_at）：
  对语义字段（logical_key + object_value + 关键 detail）取 hash，作为 `UNIQUE(logical_key, content_hash)` 去重键。
- `source_ref` ← 整条 `SourceRef` 序列化为 JSON 列（§见 schema §3）。

**append-only 写语义**（承接 schema §4a / Phase 1.5 §10.1）：
```
INSERT spine(observation_id=new, logical_key, content_hash, ...)
  ON CONFLICT(logical_key, content_hash) DO NOTHING   ← 同值重抓命中既有行、不新增
值变化 → content_hash 不同 → 新增一条 observation（v1/v2 并存）
绝不按 logical_key UPDATE 覆盖（这正是现 InMemory `bucket[fact_id]=fact` 要被纠正的点）
```

### 1.2 读取链路映射（read → 投影回领域模型）

读方法的契约**返回类型不变**（`CompanyIntelligence` / `CompanyTimelineEvent` / tuple），
SQLite 实现负责把 spine+detail **投影回** `MarketFact / FinancialFact / AnnouncementFact`：

| Repository 读方法 | SQLite 实现 |
| --- | --- |
| `get_company_intelligence_by_ts_code(ts_code)` | 经 `security.ts_code`(UNIQUE 索引) 找 security → listed_entity → company → listing/industry；按 company 取 **latest observation per logical_key**，spine JOIN detail 投影回三类 fact；`latest_quote` 见 §1.3；timeline 仍由 [timeline.py](../../services/market-data/shanhai_market_data/timeline.py) 纯函数从投影出的 fact 计算 |
| `get_company_timeline(ts_code, time_basis, latest_first)` | 同上取 fact 后调 `build_company_timeline(...)`；**timeline 永不建表**（schema §8） |
| `list_company_intelligence(limit)` | 遍历 securities → 复用 `get_company_intelligence_by_ts_code` |
| `search_company(text, limit)` | 在 `company.name/aliases` 上匹配（第一版可 `LIKE`/内存过滤，行为对齐现 InMemory）|

> **关键读语义**：默认读返回 **current truth = latest observation per logical_key**（Phase 1.5
> §10.2 的 projection）。同一 `logical_key` 多版本时取 `captured_at` 最新的一条投影成 fact。
> 「读取某 logical_key 的全部观测历史」是**新读能力**，schema §4 已明确**不在 Phase 2 加**。

### 1.3 ⚠️ Quote 双写问题（需你在本 Review 拍板）

实证（[acquisition.py](../../services/market-data/shanhai_market_data/acquisition.py#L80) /
[sync.py](../../services/market-data/shanhai_market_data/sync.py#L85)）：同一条日行情**进系统两次**：

```
1) build_quote_fact(latest_quote) → 一条 MarketFact(fact_type=QUOTE, predicate=closing_price)
   → 随 bundle 的 facts 传入 → 进 timeline（市场认知时间轴）
2) upsert_quote(latest_quote)     → 一条 QuoteSnapshot
   → 喂 read model 的 latest_quote（"当前价"，非时间轴）
```

Option C 下两种落法，**请二选一**（不改 schema，只定写链路）：

- **方案 ①（推荐）**：`upsert_quote` 写 `knowledge_observation(fact_type=quote)` + `quote_observation`
  detail（结构化 OHLCV）；`latest_quote` 读 = 该 security 下 `quote_observation` 按 `trade_date`
  最新一行。`build_quote_fact` 产出的 QUOTE MarketFact 继续走 spine（timeline 用）。
  → 统一进 spine，`latest_quote` 由 detail 表的「最新 trade_date」投影，不另立表。
- **方案 ②**：保留一张独立 `quote_snapshot` current 表（仅供 `latest_quote`），QUOTE 时间轴仍走
  spine 的 MarketFact。→ 更贴近现 InMemory 的 `_quotes_by_security`，但多一张游离于 spine 的表
  （与「统一认知时间轴」初衷有张力）。

> 我的建议是 **①**：让行情也成为 observation，`latest_quote` 退化为「detail 表最新 trade_date 投影」，
> 与 Option C「一切皆 observation」一致。若你担心首版复杂度，②也可接受。**这是本 Review 唯一的实现层
> 真实分歧，请明确。**

## 2. Identity 表确认（冻结，不变）

四层身份模型 + 全 surrogate 主键，**禁止反范式扁平表**：

```
company ──< listed_entity ──< security ──< listing      （+ industry / company_industry）
```

- ✅ 允许：`company / listed_entity / security / listing / industry / company_industry`（schema §1）。
- ❌ 禁止：`company_security`、`stock_basic_flat` 这类把多层塌缩成一张的反范式表。
- `ts_code` 永远是 `security` 的属性 + UNIQUE 索引，**永不是 identity**。
- `Security.external_ids`：**Deferred**（不进 Phase 2，schema §1）。

身份表是 **current-truth upsert**（按 surrogate PK 覆盖写），**只有 observation 是 append-only**——
两种写语义不可混淆。

## 5. 测试策略（测契约，不测 SQL）

测的是 **Repository 契约行为一致性**，不是 SQL 语句。实现结构（Step 2 落地）：

```
tests/market_data/test_repository_behavior_suite.py
    # migration-replay：一次 sync → _RecordingRepository 录写日志 →
    # 同一份 domain 写日志分别 replay 进 InMemory（reference）与 SQLite 两个 adapter，
    # 断言读模型深度相等（差异只能来自 adapter，不来自输入漂移）。
```

> 实现选择了「录制写日志 + 双 adapter 重放」而非「参数化 repo 工厂」：因为两次独立 sync 会
> 各自 mint 新 UUID（identity 漂移），无法做确定性 parity；录写日志保证两端收到同一批 domain
> 对象。这同时把 Requirement 2 的 InMemory→SQLite migration smoke test 并入同一夹具。

- **共享行为套件**对照**真实 9 方法**（upsert_* / get_* / list_* / search_*），断言
  InMemory 与 SQLite **行为一致**（写入后读出同一 `CompanyIntelligence`、timeline 同序、
  search 命中一致）。
- 额外覆盖 Phase 2 新语义：**content_hash 幂等**（同值重抓不增行）、**append-only**
  （值变化产生新 observation、读默认取最新）、**重启持久**（落盘文件重连后数据仍在）。
- SQLite 测试用 `":memory:"` 跑逻辑、用临时文件跑「重启持久」一条。
- 现有 `tests/market_data/` 既有用例必须继续全绿（业务层零改动验证）。

> 这与 RunStore 的测试范式一致（[tests/test_local_persistence.py](../../tests/test_local_persistence.py)）。
> 现有 boundary 测试（`test_repository_boundary.py` 的 isinstance + AST 禁 import DB 驱动）保持有效。

> 实现期发现并修复的关键正确性问题（非扩 schema）：身份表 writer 由 `INSERT OR REPLACE`
> 改为 `ON CONFLICT(pk) DO UPDATE`。`INSERT OR REPLACE` 在 `PRAGMA foreign_keys=ON` 下
> = DELETE 父行 + INSERT，会孤立子表 FK（二次 sync 触发 `FOREIGN KEY constraint failed`）。
> 真 UPSERT 才是「identity = current truth、原地更新」的忠实表达，恰好把 Identity / Intelligence
> 两类写语义的边界保护住。
