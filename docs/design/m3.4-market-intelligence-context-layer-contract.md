# M3.4 Market Intelligence Context Layer — Contract Design

> 状态：**🧭 Contract Design（仅设计文档；不写实现代码、不改 schema、不改 Repository 契约、不接数据源）**
> 日期：2026-06-30
> 里程碑：Milestone 3 — Market Intelligence Platform Alpha
> 前置：M3.3 Phase 2 Persistence ✅ 收口（Step 1 c0572cd / Step 2 a5629f2 / Step 3 3b148da，default backend = SQLite）
> 适用范围：**新增认知组装层的契约**（读 market-data 的知识读端口；不动 RuntimeKernel / Experience / Console）

本文件**只回答一个问题**：

> 在某个时间点，如果 AI 要理解一家公司，它**应该看到什么**——以及这个「看到的东西」**长什么形状、从哪里来、怎样可复现**。

它**不回答**「今天股价多少」。Persistence 解决「数据怎么保存」；Context Layer 解决「AI 在某个时间点知道什么」。这是 ShanHai 从「持久化基础设施」迈向「认知系统」的分水岭，也是和 daily_stock_analysis 的 `ContextPack`（今天的数据集合）最关键的分叉点。

本文是 Contract Design：定义**模型形状、组装管线、读端口契约、不变量、开放裁决项**。实现代码、模块落地、LLM 接入均**不在本阶段**。

## 1. 为什么先做 Context Layer，而不是直接写代码

M3.3 让 ShanHai 第一次拥有「可持久化认知基础设施」：

```
Market Data → Repository → SQLite        ← 只是「存东西」（Persistence）
```

但「存东西」≠「理解东西」。如果现在直接往 SQLite 里塞公司画像、行业知识、投资逻辑、市场规律，就会重蹈 daily_stock_analysis：

```
Database = 行情 + 新闻 + AI结果 + 用户行为 + 回测 + memory   ← 大杂烩
```

Context Layer 的职责是把「散落的 observation」组装成「某个时间点的、可被 AI 推理消费的市场认知状态」：

```
Knowledge Object  →  Context Snapshot  →  Reasoning Input  →  LLM
   （主体聚合）        （as_of 认知状态）      （LLM 形状）       （推理）
```

**这一层定义对了，后面 iFinD / 免费行情 / LLM / Agent 都只是接入问题；定义错了，再多数据也只是一个更大的 daily_stock_analysis。**

## 2. 命名与边界澄清（架构观察，需确认）

### 2.1 ⚠️ 与既有 `RuntimeContext` 的命名冲突

代码库**已经有**一个 `RuntimeContext`（`services/runtime-kernel`，见
[runtime-context-v1-final-contract-review.md](./runtime-context-v1-final-contract-review.md)）：它是 **Agent 执行初始化快照**（identity/task/intent/experience/policy/constraint/environment 七段，`frozen + extra=forbid`），描述「一次 Agent 运行如何被装配」。

> 关系：未来 Agent 推理时，`MarketContextSnapshot` 会成为喂给 RuntimeContext 的 **task/intent 输入素材之一**，但二者是不同层、不同生命周期、不同 owner。

## 3. 认知分层：Observation ≠ Knowledge ≠ ContextSnapshot

这是本设计的骨架，也直接回答评审 checkpoint 2（区分 observation 与 knowledge）：

```
┌────────────────────────────────────────────────────────────────────┐
│ Observation  （已存在，M3.3 append-only spine）                      │
│   一条不可变记录：「source X 在 captured_at T 报告：主体的某属性 = V」 │
│   带 provenance（SourceRef）/ confidence；可与其它源冲突；保留历史    │
├────────────────────────────────────────────────────────────────────┤
│ Knowledge    （派生的「当前信念」）                                   │
│   对某 logical_key 的多条 observation 调和后的最优理解                │
│   M3.3 读投影 = latest per logical_key（current truth）              │
├────────────────────────────────────────────────────────────────────┤
│ KnowledgeObject （主体聚合，M3.5 细化）                              │
│   以 company / industry / event 为中心，把多个 Knowledge 聚成画像     │
├────────────────────────────────────────────────────────────────────┤
│ MarketContextSnapshot （本层产物）                                    │
│   在指定 as_of 把 KnowledgeObject「冻结」为一份可推理、可复现的认知态  │
└────────────────────────────────────────────────────────────────────┘
```

关键不变量：
- **Observation 是事实输入，Knowledge 是派生信念**。Context Layer 永不修改 observation；它**只读、只投影**。
- 同一 logical_key 的多条 observation（不同 `content_hash` / 不同 `captured_at`）在 M3.3 已 append-only 保留（Step 2 已验证 SQLite retains history）。这是 Context Layer 做 bitemporal 的**底料**。

## 4. 核心契约：`MarketContextSnapshot`

### 4.1 设计原则（每一条都是「不是 ContextPack」的护栏）

| # | 原则 | 与 daily_stock_analysis `ContextPack` 的区别 |
| - | ---- | ------------------------------------------ |
| P1 | **必须带 `as_of`（双时间轴）** | ContextPack 无时间身份，是「今天拿到的一坨」 |
| P2 | **必须可复现**：相同 `(subject, as_of)` → 相同 snapshot | ContextPack = 「API 今天返回什么就是什么」，不可复现 |
| P3 | **组装期不调用 provider**：只读已持久化的 observation | ContextPack = 「现在去把所有数据抓一遍」 |
| P4 | **每个数值都带 provenance + confidence + 时间** | ContextPack 只有裸值 |
| P5 | **携带 `data_quality` 与 `historical_cognition`** | ContextPack 没有「认知质量」与「我以前怎么想的」 |

> P3 是最硬的护栏：**Context Layer 依赖的是「我们已经知道并持久化的东西」，不是「现在能抓到什么」。** 数据进入系统是 Acquisition Layer 的事（上游、异步、带 SourceRef）；Context Layer 永远在「已沉淀的 observation」上做投影。

### 4.2 Snapshot 契约形状（contract sketch，非实现）

字段命名尽量复用已存在的领域模型（`SourceRef` / `SubjectRef` / `FactType` / `TimeBasis` / `FinancialFact` / `MarketFact` / `AnnouncementFact` / `QuoteSnapshot`），避免另造平行词汇：

```jsonc
{
  "subject": {                      // 复用 SubjectRef 语义（surrogate id，非 ts_code 身份）
    "entity_type": "company",
    "entity_id": "<surrogate company_id>",
    "ts_code": "600519.SH",         // 仅作为属性/检索键，永不是身份
    "label": "贵州茅台"
  },

  "as_of": {                        // P1：双时间轴（bitemporal），见 §5
    "effective_at": "2026-06-30T14:00:00Z",   // 世界视角：截至这一刻「成立/可见」的事实
    "knowledge_at": "2026-06-30T14:00:00Z"     // 系统视角：只采纳 captured_at ≤ 此刻 的 observation
  },

  "identity": {                     // 当前真相（current-truth upsert 来源）：company/industry/listing
    "company_profile": { "value": {}, "provenance": [], "confidence": 1.0 },
    "industry":        { "value": {}, "provenance": [], "confidence": 1.0 },
    "listing":         { "value": {}, "provenance": [], "confidence": 1.0 }
  },

  "financial_state": {              // 投影自 FinancialFact（report_period/metric/unit/yoy/qoq）
    "latest_report": { "report_period": "", "metrics": [], "provenance": [] },
    "growth_trend":  { "series": [], "provenance": [] },
    "profit_quality":{ "metrics": [], "provenance": [] }
  },

  "market_state": {                 // 投影自 QUOTE observation（latest as-of effective）
    "price":       { "value": null, "trade_date": null, "provenance": [] },
    "valuation":   { "metrics": [], "provenance": [] },
    "relative_strength": { "metrics": [], "provenance": [] }
  },

  "events": [                       // 投影自 timeline（CompanyTimelineEvent），仅取 as_of 之前
    { "event_id": "", "event_time": "", "event_time_basis": "published_at",
      "event_type": "announcement", "title": "", "summary": "",
      "fact_refs": [], "provenance": [] }
  ],

  "historical_cognition": {         // P5：我以前怎么想、预测、被验证（引用而非复制）
    "previous_analysis":   [ { "ref": "<artifact/run id>", "at": "" } ],
    "previous_prediction": [ { "ref": "", "horizon": "", "at": "" } ],
    "feedback":            [ { "ref": "<outcome/lesson id>", "at": "" } ]
  },

  "data_quality": {                 // §6：这份认知本身有多可信/多完整/多新鲜
    "coverage": { "identity": 1.0, "financial": 0.0, "market": 0.0, "events": 0.0 },
    "freshness": { "max_staleness_days": null, "stale_sections": [] },
    "conflicts": [ { "logical_key": "", "competing_sources": [] } ],
    "trust_floor": "L1_official",   // 本 snapshot 内最低 trust_level
    "missing": [ "financial_state.profit_quality" ]
  },

  "snapshot_meta": {
    "snapshot_id": "<deterministic from subject+as_of+source set>",
    "schema_version": "market_context_snapshot.v1",
    "assembled_at": "2026-06-30T14:01:10Z",
    "reproducible_from": { "observation_cutoff": "knowledge_at" }
  }
}
```

> 注意：上面每个 leaf 都是 **`{ value, provenance[], confidence }`** 三元组形状，而不是裸值——这是 P4 的结构化体现。`provenance[]` 元素即现有 `SourceRef`（provider/dataset/captured_at/raw_snapshot_ref/version/hash/trust_level），**零新造 provenance 词汇**。

### 4.3 组装管线（contract，非实现）

```
                     ┌─────────────────────────────────────────────┐
   (read port)       │  ContextAssembler.assemble(subject, as_of)    │
 KnowledgeReadPort ──▶│   1. 读 as_of 之前的 observation（bitemporal）│
   （§4.4 新增端口） │   2. 调和为 Knowledge（latest as-of）         │
                     │   3. 聚合为 KnowledgeObject（subject 中心）    │
                     │   4. 计算 data_quality（coverage/freshness…） │
                     │   5. 冻结为 MarketContextSnapshot             │
                     └───────────────────────┬─────────────────────┘
                                              │
                                              ▼
                            ReasoningInputBuilder（M3.7）
                                              │
                                              ▼
                                             LLM
```

- ContextAssembler **只依赖一个读端口**（§4.4），不依赖 provider、不依赖 SQLite 具体实现、不依赖 LLM。
- `MarketContextSnapshot → ReasoningInput`（LLM prompt 形状）是**下一层（M3.7）**的事，本文只冻结到 Snapshot。

### 4.4 它需要的读契约：**additive read port（不改冻结的 9 方法）**

M3.3 的 9 方法读投影是 **latest-only / current-truth**（schema 明确把「observation-history 新读方法」列为 Phase 2 不做）。Context Layer 要做 `as_of` 认知，需要**「截至某 knowledge_at 的 observation」**——这是一个**新读能力**。

> **决定（待裁）**：新增一个**独立只读端口** `KnowledgeReadPort`（暂名），**不污染** `MarketKnowledgeRepository` 的 9 方法。形状大致：
>
> ```
> get_observations_as_of(subject, *, knowledge_at, effective_at=None, fact_types=()) -> tuple[Observation, ...]
> ```
>
> - 它读的是 append-only spine（M3.3 已保留历史），按 `captured_at ≤ knowledge_at` 过滤。
> - 既有 9 方法 = 「current truth」读；新端口 = 「historical cognition」读。两者并存、互不修改。
> - 端口可由现有 SQLite adapter **额外实现**（未来实现期），InMemory reference adapter 同样可实现 → 仍可 parity 测试。
>
> 三个落点选项见 §10-O1。

## 5. Bitemporal 模型（核心，回答 checkpoint 1 & 4）

### 5.1 两条时间轴

ShanHai 的 fact 模型**已经**自带三个时间戳（M2.5 冻结，永不塌缩）：

| 时间戳 | 轴 | 含义 |
| ------ | -- | ---- |
| `occurred_at` | **effective / valid time** | 事件在世界里真正发生 |
| `published_at` | **effective / valid time** | 市场可见（披露）时刻 |
| `captured_at` | **knowledge / transaction time** | ShanHai 学到这件事的时刻 |

`MarketContextSnapshot.as_of` 因此是**双轴**：
- `effective_at`：截至世界的哪一刻（用 occurred_at/published_at 过滤/排序事件）。
- `knowledge_at`：系统只采纳 `captured_at ≤ knowledge_at` 的 observation。

> 这正是「**what did we know at that time?**」——也是上一轮 review 反复强调、且被 M3.3 Step 2「SQLite retains observation history」验证保护住的护城河。

### 5.2 as_of 语义（首版）

```
回答「2026-06-30 14:00 AI 理解贵州茅台应看到什么」=
  取所有 captured_at ≤ knowledge_at 的 observation
  → 按 logical_key 取 effective time ≤ effective_at 中最新的一条
  → 聚合 + 质量评估 + 冻结
```

- 若 `knowledge_at = now`：退化为「当前认知」（≈ 现 9 方法的 current truth，但带 quality/provenance/历史认知）。
- 若 `knowledge_at` 设为过去某刻：得到「**当时 AI 会看到的认知**」——历史认知回放（checkpoint 4）。因为 observation append-only + snapshot 可复现（P2），回放是**确定性**的，不需要保存当时的快照副本。

### 5.3 本阶段对 bitemporal 的诚实边界

- **首版只需 `knowledge_at` 过滤 + effective 排序**即可支撑「认知回放」，**不要求**完整 SQL:2011 bitemporal 表/区间合并。
- 既有 9 方法**不加 `as_of`**（保持冻结）；bitemporal 能力完全落在 §4.4 的新读端口。
- ContextSnapshot **是否落库持久化**留作开放项（§10-O2）：首版倾向**按需计算（不落表）**，因为 append-only + 可复现意味着「快照可随时重算」，落表是优化而非正确性需求。

## 6. Data Quality / Provenance 面（回答「认知有多可信」）

daily_stock_analysis 的 ContextPack 只有数据，没有「这份认知本身的质量」。ShanHai 的 snapshot 必须自带 `data_quality`：

- **coverage**：identity/financial/market/events 各维度是否有数据（缺失要显式 `missing[]`，不能静默当 0）。
- **freshness**：各 section 的最大滞后（`captured_at` 距 `as_of` 多久），标出 `stale_sections`。
- **conflicts**：同一 logical_key 多源不一致时，列出 `competing_sources`（不静默取一个）。
- **trust_floor**：本 snapshot 内最低 `trust_level`（L1_official / L3_public_aggregator…），让推理层知道「这份认知最弱的一环来自哪种源」。

> 这让 LLM/Agent 在推理时能区分「我知道且可信」「我知道但存疑」「我不知道」——这是认知系统与数据堆的本质差异。Provenance **直接复用 `SourceRef`**，不另造。

## 7. Provider 无感知保证（回答 checkpoint 3：未来 iFinD/Wind）

Context Layer **不知道、也不应该知道**数据来自 akshare / iFinD / Wind / Tushare Pro。它只读 `KnowledgeReadPort` 给出的 observation，每条带 `SourceRef`。

```
            Acquisition Layer (M3.6，未来)             Context Layer (本文)
  akshare ─┐
  iFinD  ──┤→ Provider Interface → upsert_* → Repository → KnowledgeReadPort → ContextAssembler
  Wind   ──┘   （带 SourceRef）       (M3.3 已冻结契约)        (§4.4 新端口)
```

- 接入 premium provider = 在 **Acquisition Layer** 多产生带 `SourceRef(provider=ifind, trust_level=…)` 的 observation；ContextSnapshot 自动经 provenance/trust_floor 反映，**Context 层零改动**。
- `SHANHAI_DATA_MODE = free | premium | mixed` 是 **Acquisition 层装配开关**，对 Context Layer **不可见**。
- 反模式（禁止）：`if source == "ifind": ...` 出现在 Context/Repository 层。正确：`repository.upsert_*(fact)`，源信息只在 `SourceRef` 里。

> **本阶段不接 iFinD**。原因：现在还不知道 Context Layer 需要哪些字段。先定义「需要什么认知 → 需要什么数据 → 需要什么 provider」，不要反过来「有什么 API → 往里塞」。iFinD token 等到「财务历史导入 / 公司画像初始化 / 产业链 / 一致预期 / 历史事件库」阶段再接。

## 8. 评审 5 个 checkpoint 的逐条设计回应

| # | Checkpoint | 本设计如何满足 |
| - | ---------- | ------------- |
| 1 | ContextSnapshot 是否支持 bitemporal | ✅ `as_of = {effective_at, knowledge_at}` 双轴（§5）；底料是 M3.3 append-only observation + 三时间戳；首版用 knowledge_at 过滤 + effective 排序，不强求完整区间 bitemporal |
| 2 | 是否区分 observation 与 knowledge | ✅ 四层分层（§3）：Observation（不可变输入）→ Knowledge（派生信念）→ KnowledgeObject（主体聚合）→ ContextSnapshot（as_of 冻结）；Context 层只读不改 observation |
| 3 | 是否支持未来 iFinD/Wind provider | ✅ Context 层对 provider 无感知（§7）；接入只在 Acquisition 层增 observation+SourceRef；`SHANHAI_DATA_MODE` 不进 Context 层；禁 `if source==...` |
| 4 | 是否支持历史认知回放 | ✅ 设过去的 `knowledge_at` 即「当时认知」；append-only + 可复现（P2）→ 回放确定性；`historical_cognition` 块引用过往 analysis/prediction/feedback（引用非复制） |
| 5 | 是否避免变成 ContextPack | ✅ 5 条护栏（§4.1 P1–P5）：带 as_of / 可复现 / 组装期不抓数据 / 每值带 provenance+confidence / 带 data_quality+historical_cognition。ContextSnapshot 是「认知状态」，不是「今天的数据集合」 |

## 9. 与既有架构的依赖边界（必须遵守）

- `services/market-data/**` 现有**依赖禁令**：禁 import runtime/experience/memory/evolution/feedback，禁 trading 术语（`test_dependency_boundary` AST 校验）。
- 因此 `historical_cognition`（previous_analysis/prediction/feedback）**只能引用 id/ref**，不能 import experience 模块、不能内嵌其内容——与 Experience 层「引用而非复制」「append-only」「Agent 只读」诸不变量一致。
- **模块落点是开放项**（§10-O3）：若 Context Layer 需要同时触达 market-data 知识 + experience 历史认知，它应是 market-data **之上的新模块**，而非塞进 market-data（后者受依赖禁令约束，且职责是 persistence/knowledge-fact，不该承担跨域认知组装）。
