## 0. 本设计的硬约束（不变量，继承 ADR 0020 §4/§5）

- **doc-only**：本文不写任何 `.py`，不建 `services/market-intelligence/knowledge` 目录，不建测试文件。
- **不接 LLM**：pipeline 的 reasoning 步只定义 `ReasoningPort` 抽象（Protocol）；具体实现 + LLM 接入属 reasoning-engine（M3.7）。S4.0 参考实现＝deterministic `NoopReasoner`。
- **不改 schema**：不新增 `knowledge_object` / `knowledge_revision` / `observation_history` 表。KnowledgeObject 首版**不落库**（对齐 ADR 0019 D6：deterministic view，落表是优化非正确性需求）；版本链在内存/引用层表达。
- **不推翻 0019**：deterministic Knowledge（D4）与 deterministic ContextAssembler（R1-4）**保持冻结**；本层是 additive 并行解释层（ADR 0020 D2）。

## 1. 概念地基：为什么 Knowledge 必须与 Observation 分层

ADR 0019 已冻结「Observation = 事实，Knowledge（latest per key）= 事实层当前真相」。S4.0 引入的是**第三种东西**——**解释（interpretation）**：

```
Observation（事实）          2026-06-01：公司发布公告，内容 = X          ← 可确定性回放
      │
      ├─(A) latest per key ─► Knowledge（事实当前值）  最新披露 = X           ← deterministic（0019 D4）
      │
      └─(B) 解释性演化 ─────► KnowledgeObject（信念）  「该公告提高盈利确定性」  ← 经 reasoning，本层
                                                        「市场认知 成长股→现金流资产」（跨年演化）
```

三者关系的判定标准（冻结）：

| | 生成方式 | 可否 deterministic 复现 | 是否需推理 | 是否演化 |
|--|--|--|--|--|
| Observation | 数据流入 append | 是（append-only spine） | 否 | 否（不可变） |
| Knowledge（0019 D4） | latest per logical_key | 是（确定性投影） | 否 | 随 observation 变（无版本历史） |
| **KnowledgeObject（本层）** | Evolution Pipeline | 是（版本链 + evidence 回放） | **是**（经 ReasoningPort） | **是**（v1→v2→v3 版本链） |

**核心分水岭**：`Observation ≠ Current State ≠ Interpretation`。daily_stock_analysis 把三者塌缩成「今天的数据 → LLM → AnalysisHistory（覆盖）」；ShanHai 保持三层分立，且解释层每条信念绑定证据、版本链演化。

## 2. 交付 ①：Knowledge Object Contract

`KnowledgeObject` 是**以 subject 为中心的解释性认知资产**——一组 evidence-bound、versioned 的信念，**不是** AI summary storage。字段树（对齐用户冻结形状 + ADR 0017 Candidate 数据模型的结构化/来源-证据分离/版本-血缘精神）：

```python
# contract sketch —— 非实现；所有 model frozen=True, extra="forbid"，集合用 tuple
# 属 market-intelligence（knowledge 子域，本期只登记不建目录）

class Belief(_FrozenModel):
    belief_id: str                          # 稳定标识（可被 revision / snapshot 引用）
    statement: BeliefStatement              # 结构化信念主张（非自由文本 summary，见下）
    evidence_refs: tuple[EvidenceRef, ...]  # 支撑此信念的 Observation 引用（必须非空，见 §4）
    confidence: float                       # 0~1，随证据演化（非静态 occurrence）
    polarity: BeliefPolarity                # supportive / cautionary / neutral（立场）
    scope: BeliefScope | None = None        # 适用边界（何时成立 / 何时失效，对齐 ADR 0017 applicability）

class BeliefStatement(_FrozenModel):        # 结构化，杜绝退化成 LLM Summary
    dimension: str                          # 信念维度（如 profitability / moat / valuation_regime）
    claim: str                              # 结构化主张（可枚举/受控词表，非任意长文）
    # summary: str | None                   # 若保留仅作人类可读派生视图，不作语义/匹配依据

class KnowledgeObject(_FrozenModel):
    subject: SubjectRef                     # import 自 market-data（下游方向，允许）
    version: int                            # 版本号 v1=1, v2=2 …（见 §3）
    previous_version: KnowledgeObjectRef | None = None   # 版本链锚点；v1 为 None（见 §3）
    beliefs: tuple[Belief, ...] = ()        # 解释性信念集合（每条 evidence-bound）
    evidence_refs: tuple[EvidenceRef, ...] = ()          # 对象级证据索引 = 各 belief evidence 的并集（引用）
    confidence: float = 0.0                 # 对象级综合置信度（派生自 beliefs，演化量）
    as_of_knowledge_at: datetime            # 本版本采纳的 observation 截止线（对齐 0019 D5 knowledge_at）
    updated_at: datetime                    # 本版本生成时刻
    # ── 可复现 meta（非认知内容；确定性锚点）──
    object_id: str                          # deterministic from (subject) —— 跨版本稳定，标识"同一主体的认知"
    revision_id: str                        # deterministic from (object_id, version, evidence 身份集)
    schema_version: Literal["s4.0"] = "s4.0"
```

铁律（本 contract 的护栏）：

- **belief 必须有 evidence**（`Belief.evidence_refs` 非空）：这是构造前置条件，也是本层与 AI summary storage 的唯一硬边界。无证据信念 = 幻觉，禁止落地（见 §4）。
- **`object_id` 跨版本稳定，`revision_id` 每版本唯一**：`object_id` 标识「关于该 subject 的认知」这条演化线；`version + previous_version` 串起版本链；`revision_id` 是某一版本的确定性指纹。
- **结构化信念而非 summary**：`BeliefStatement.claim` 是结构化/受控主张；`summary` 若保留仅人类可读派生视图（继承 ADR 0016/0017/0018「name/summary 不作语义载体」）。
- **引用而非复制**：`evidence_refs` 只持 Observation 身份，不内嵌 observation value（见 §4）。
- **对象级 `confidence` 派生自 beliefs**，是演化量（区别于 observation 的静态 confidence）。

> **不采纳形状**（用户明确否定）：`CompanyKnowledge { name, industry, valuation, summary }`——它会变成 AI summary storage / 更贵的 daily_stock_analysis。

## 3. 交付 ②：Knowledge Revision Model（版本链，禁覆盖）

演化是**新增版本（append）**，不是原地修改（overwrite）。对齐用户「v1 → v2 → v3，而非 `.update()`」与 ShanHai append-only 精神。

```
object_id = KO(贵州茅台)
   │
   ├── v1 (revision_id=r1)  beliefs=[成长股, ROE 30%+]                    evidence=E1
   │      ▲ previous_version = None
   │      │
   ├── v2 (revision_id=r2)  beliefs=[增速下降但现金流稳, ROE 维持]        evidence=E2
   │      ▲ previous_version → v1(r1)
   │      │
   └── v3 (revision_id=r3)  beliefs=[稳定现金流资产, 风险=消费降级]        evidence=E3
          ▲ previous_version → v2(r2)
```

```python
# contract sketch —— Revision 是"从 vN 到 vN+1 的一次演化事件"，本身也 append-only
class KnowledgeObjectRef(_FrozenModel):     # 指向某个版本（版本链锚点）
    object_id: str
    version: int
    revision_id: str

class KnowledgeRevision(_FrozenModel):      # 一次演化的记录（不可变）
    revision_id: str                        # = 新版本的 revision_id
    object_id: str
    from_version: int | None                # None = 初次创建（v1）
    to_version: int                         # from_version + 1（或 1）
    triggered_by: CandidateRef              # 哪个 Candidate Knowledge Change 触发（来源，见 §5）
    reasoning_ref: str | None = None        # ReasoningPort 产出的推理记录引用（M3.7 落地；S4.0 = None）
    belief_delta: BeliefDelta               # 本次相对上一版本的信念变化（added / revised / retired）
    created_at: datetime

class BeliefDelta(_FrozenModel):            # 显式表达"改了什么"，可追溯
    added: tuple[str, ...] = ()             # 新增 belief_id
    revised: tuple[str, ...] = ()           # confidence/scope 变化的 belief_id
    retired: tuple[str, ...] = ()           # 不再持有的 belief_id（retire 非 delete：旧版本仍保留）
```

铁律：

- **禁覆盖、禁删除旧版本**：v1/v2 永久保留。retire 一条 belief 只是「新版本不再包含它」，旧版本里它仍在——因此可回放「AI 在 2026-03 为什么持有该信念」。
- **Revision 是唯一演化通道**：任何 KnowledgeObject 版本的产生都必须经一次 `KnowledgeRevision`（经 Revision Gate，见 §5）；禁止直接构造 vN+1 绕过 pipeline。
- **`belief_delta` 显式化**：不靠 diff 两个版本反推，而是 Revision 自身记录 added/revised/retired，保证演化历史可读、可审计。
- **版本链 + bitemporal 复现（对齐 0019 D5）**：给定 `knowledge_at`，可确定性定位「当时生效的 KnowledgeObject 版本」——`version` 链 + `as_of_knowledge_at` 单调，选 `as_of_knowledge_at ≤ knowledge_at` 的最大版本。

## 4. 交付 ③：Evidence Binding Contract（belief ↔ Observation）

每条 belief 必须绑定到 append-only spine 的具体 Observation 行——这让信念**可追溯、可证伪、可确定性回放**。对齐 ADR 0017 Decision C「`source_refs` 与 `evidence_refs` 分离」。

```python
# contract sketch —— EvidenceRef 锚定一条 observation 的身份（不内嵌值）
class EvidenceRef(_FrozenModel):
    logical_key: str            # observation 身份的一半（= 旧 fact_id / lineage）
    content_hash: str           # observation 身份的另一半（幂等锚点）
    captured_at: datetime       # 便于版本链复现/新鲜度排序，不是 value 本身
    relation: EvidenceRelation  # supports / contradicts / contextualizes（证据对信念的作用）
```

绑定语义（冻结）：

```
Belief ──evidence_refs[]──► EvidenceRef ──(logical_key, content_hash)──► Observation（market-data spine）
                                                                               ▲
                            KnowledgeObject 引用 Observation，单向、只读 ────────┘
                            Observation 永不反向知道 KnowledgeObject（0019 R1-1 铁律）
```

- **来源 vs 证据分离**（对齐 ADR 0017）：
  - **证据（evidence_refs）**：支撑/证伪信念的 observation，生命周期 = 随版本链持续累积（`relation` 可为 `contradicts`——反例也是证据）。
  - **来源（source）**：提出此次修订的依据 = `KnowledgeRevision.triggered_by`（哪个 Candidate 触发），生命周期 = 提出修订时。
- **可确定性回放**：因 `EvidenceRef` 锚定 append-only spine 行，给定 `knowledge_at` 可经 `ObservationReadPort.query(subject, knowledge_at=...)` 重放「该信念当时依据了哪些 observation」——S1–S3 的 observation 回放能力在此被 belief 复用。
- **禁内嵌 observation 值**：`EvidenceRef` 只持身份 + `captured_at`，不复制 `object_value`（避免事实与解释双写不一致；值经 `ObservationReadPort` 按需取）。
- **evidence 非空校验发生在 Revision Gate**（见 §5），是 KnowledgeObject 落地的强制门槛。

## 5. 交付 ④：Evolution Pipeline（Observation → Candidate → reasoning → Revision）

LLM 进入认知系统的**唯一合法路径**（ADR 0020 D6）。镜像 ADR 0016/0017 的 `Event → Candidate → Validation → Promotion → Artifact`，把 LLM 严格夹在中间推理步、经 `ReasoningPort` 注入。

```
[1] Observation（append-only 事实，经 ObservationReadPort 只读）
        │  ChangeDetector：检测与现有 belief 相关的 新证据 / 冲突 / 缺口（deterministic）
        ▼
[2] CandidateKnowledgeChange（结构化"信念修订候选"= 可验证假设，非 summary）
        │  ReasoningPort.reason(candidate, evidence) —— LLM 作 reasoner（M3.7；S4.0=NoopReasoner）
        ▼
[3] ProposedRevision（拟修订：拟 added/revised/retired beliefs + 各自 evidence_refs）
        │  RevisionGate.admit(proposed) —— 唯一写入口：校验 evidence binding，只固化不创作
        ▼
[4] KnowledgeRevision → KnowledgeObject@vN+1（append，evidence-bound，禁覆盖 vN）
```

每步契约草图与边界：

```python
# [1]→[2] ChangeDetector：deterministic，不含 LLM（对齐 0019 R1-4 精神）
class ChangeDetector(Protocol):
    def detect(
        self, subject: SubjectRef, current: KnowledgeObject | None,
        observations: tuple[Observation, ...],      # 经 ObservationReadPort 取
    ) -> tuple["CandidateKnowledgeChange", ...]: ...

class CandidateKnowledgeChange(_FrozenModel):        # 镜像 ExperienceCandidate（ADR 0017 Decision C）
    candidate_id: str
    subject: SubjectRef
    change_kind: ChangeKind                          # new_evidence / conflict / gap
    hypothesis: RevisionHypothesis                   # 结构化假设（拟修订什么信念、为何）——非 summary
    source_refs: tuple[EvidenceRef, ...]             # 触发此候选的 observation（来源）
    evidence_refs: tuple[EvidenceRef, ...]           # 可用于验证的 observation（证据），与 source 分离
    hypothesis_version: int                          # 假设版本（对齐 0017：候选自身可演化）
    created_at: datetime

# [2]→[3] ReasoningPort：LLM 唯一入口，只留接口（ADR 0020 D3）；不 import reasoning-engine
class ReasoningPort(Protocol):                       # 由 reasoning-engine（M3.7）实现；S4.0 用 NoopReasoner
    def reason(
        self, candidate: CandidateKnowledgeChange,
        evidence: tuple[Observation, ...],           # 已解析的证据（经 ObservationReadPort）
    ) -> "ProposedRevision": ...
    # 铁律：reason 消费 候选+证据，产出"拟修订提案"；不写 KnowledgeObject、不生成 evidence、
    #       产出的每条拟 belief 必须携带 evidence_refs（否则被 [3] RevisionGate 拒绝）

class NoopReasoner:                                  # S4.0 deterministic 参考实现（对齐 ADR 0017 NoopValidator）
    def reason(self, candidate, evidence):
        # 不推理、不总结：把 candidate.hypothesis 原样映射为 proposed（evidence 直通），供 parity/契约测试
        ...

# [3]→[4] RevisionGate：KnowledgeObject 唯一写入口（对齐 ADR 0017 Promotion Gate / 0018 ArtifactBuilder）
class ProposedRevision(_FrozenModel):
    candidate_id: str
    object_id: str
    proposed_beliefs: tuple[Belief, ...]             # 每条必须 evidence_refs 非空
    proposed_delta: BeliefDelta
    reasoning_ref: str | None = None                 # M3.7 落地；S4.0 = None

class RevisionGate(Protocol):
    def admit(self, current: KnowledgeObject | None, proposed: ProposedRevision) -> KnowledgeRevision: ...
    # 铁律（"只固化不创作"，继承 ADR 0017 Decision E / 0018 D4）：
    #   - 校验每条 proposed_belief.evidence_refs 非空且指向真实 observation（经 ObservationReadPort 存在性）
    #   - 只把已校验的 proposed 固化为 vN+1，不新增/改写信念内容
    #   - 禁携带 LLM 自由文本作为信念内容载体（禁 LLM Summary 作 Gate 内容生成器）
    #   - 是唯一写入口：禁任何路径绕过 Gate 直造 KnowledgeObject 版本
```

铁律（pipeline 级）：

- **LLM 只在 [2]→[3] 且经 `ReasoningPort`**：禁 `Observation → LLM → Knowledge` 直连（ADR 0020 D6）。
- **`ChangeDetector` 与 `RevisionGate` 均 deterministic、不含 LLM**（推理只在中间步），守住「事实检测/证据校验可确定性复现」。
- **候选 ≠ 修订**：`CandidateKnowledgeChange` 是假设，`KnowledgeRevision` 是经证据校验后固化的版本；禁跳步。
- **S4.0 用 `NoopReasoner`**：全 pipeline 可用 deterministic 参考推理器跑通契约测试，**不接任何 LLM**。

## 6. 交付 ⑤：与 MarketContextSnapshot 的边界

冻结两层职责，防止 Snapshot 成为 Evolution 的旁路，也防止 Evolution 污染 deterministic 的 Snapshot（守 ADR 0019 R1-4）。

| | Knowledge Evolution（本层 / ADR 0020） | MarketContextSnapshot（ADR 0019 D3/D6） |
|--|--|--|
| 职责 | **生产**演化的解释性信念（版本链） | **引用**某 as_of 冻结的认知态（ref-based） |
| 时间性 | 跨时间演化 v1→v2→v3 | 单点冻结（as_of 一刻） |
| 含推理？ | 是（经 ReasoningPort，M3.7） | **否**（deterministic assembly，R1-4） |
| 可变？ | append 新版本 | 不可变，deterministic view（不落库，D6） |
| 谁读谁 | 读 Observation，产出 KnowledgeObject | 读 KnowledgeObject 的**特定版本**（ref） |

数据流边界：

```
Observation ──► [Knowledge Evolution: Candidate→reasoning→Gate] ──► KnowledgeObject@vN
                                                                          │
                                                              (确定性选版本 by knowledge_at)
                                                                          ▼
                       ContextAssembler(subject, as_of) ──引用──► MarketContextSnapshot.knowledge_refs
                       （纯 deterministic，R1-4，不触发推理）
```

- **Snapshot 引用 KnowledgeObject 特定版本**：`MarketContextSnapshot.knowledge_refs`（ADR 0019 §4 已定义 `KnowledgeRef`）指向某 `object_id@version`（经 `revision_id` 锚定），**ref-based 不内嵌信念值**。
- **ContextAssembler 保持纯 deterministic**：给定 `as_of.knowledge_at`，确定性选出「当时生效的 KnowledgeObject 版本」（§3 复现规则）并引用；**永不触发/内嵌** Evolution 推理（守 R1-4）。
- **Evolution 永不读/写 Snapshot**：pipeline 只消费 Observation、产出 KnowledgeObject Revision；Snapshot 是下游只读消费者。
- **`KnowledgeRef` 扩展点（待 0019 侧实现期对齐）**：现有 `KnowledgeRef{logical_key, resolved_from, confidence}` 面向 deterministic Knowledge（latest per key）；引用 interpreted KnowledgeObject 时需能锚定 `object_id@version/revision_id`。此为实现期在 0019 与本契约间对齐的**开放点**（O1，见 §8），S4.0 只记录不改 0019 冻结形状。

## 7. 模块归属与依赖方向（登记，不建目录）

> **命名同步（R4）**：S4.0 Review 指定子域布局为 `market-intelligence/evolution/`（含 Candidate / Revision / KnowledgeObject）。本节原登记为 `knowledge/`，现**收敛为 `evolution/`**；authoritative 目录布局见 [S4.1 Implementation Design §1](./s4.1-knowledge-evolution-implementation-design.md)。

```
services/
├── market-data              Observation / spine / ObservationReadPort（M3.3+S3，冻结）
├── market-intelligence
│   └── shanhai_market_intelligence/
│       ├── ...（Context 侧：models / assembler / quality，M3.4 S1–S3）
│       └── evolution/        ← 本层登记位（S4.0 只登记，不建目录、不写代码；布局见 S4.1 §1）
│           # KnowledgeObject / Belief / EvidenceRef / KnowledgeRevision
│           # ChangeDetector / RevisionGate / ReasoningPort(Protocol) / NoopReasoner
├── reasoning-engine         ReasoningPort 实现 + LLM（M3.7，只登记不建）
└── runtime-kernel

# 不变量（沿用 ADR 0019 R1 + ADR 0020 §5 + D9/R4-1、R4-2）
- 依赖箭头永不反向；market-data 永不 import intelligence 概念（R1-1）
- evolution/ 读 Observation 只经 ObservationReadPort（基元签名）
- evolution/ 不 import reasoning-engine；只依赖 ReasoningPort（Protocol）
- evolution/ MUST NOT import / 依赖 MarketContextSnapshot（D9，R4-1）；Context MAY consume Knowledge
- LLM 唯一入口 = pipeline reasoning 步（经 ReasoningPort）；禁 Observation→LLM→Knowledge 直连
- KnowledgeObject 唯一写入口 = RevisionGate；beliefs 必须 evidence-bound
- RevisionGate 只做 deterministic 校验（evidence/schema/version/provenance），禁 reasonableness/quality 判断（R4-2）
- 版本链 append-only：禁覆盖 / 禁删除旧版本
- Observation 永不反向知道 KnowledgeObject；Snapshot 永不触发 Evolution 推理
```

依赖边界未来由 AST 校验守护（延续 market-data/market-intelligence 的 `test_dependency_boundary.py` 模式）：`evolution/` 禁 import `reasoning-engine` 具体实现、禁 import `MarketContextSnapshot`（D9）、禁 trading 术语、只经 `ObservationReadPort` 触达 market-data。
